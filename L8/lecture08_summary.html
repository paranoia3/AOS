Advanced Operating Systems · Lecture 8

File System Internals

Внутреннее устройство файловых систем — резюме и пояснения

Эта лекция углубляется во внутреннее устройство файловых систем: типы ФС и тома, разделы и загрузочные блоки, монтирование и проверка целостности, виртуальная файловая система (VFS) как объектно-ориентированный слой над разными ФС, удалённые ФС и их аутентификация, модели семантики согласованности (UNIX / session / immutable-shared-files) и сетевая ФС Sun NFS — её протоколы mount и собственно NFS, трансляцию путей, кэширование атрибутов и блоков, stateless-семантику.

1. Цели лекции и общая картина

  • ОС общего назначения предоставляют много типов ФС — от специальных до универсальных.
  • Тома с файловыми системами могут быть смонтированы в общее пространство файловой системы компьютера.
  • В зависимости от ОС, это пространство либо «бесшовное» (примонтированные ФС встраиваются в общее дерево каталогов, как в UNIX), либо «различимое» (у каждой ФС свой идентификатор, как буквы дисков в Windows).
  • Хотя бы одна ФС должна быть загрузочной — содержать ОС. Сначала запускается boot loader: простая программа, умеющая найти ядро в ФС, загрузить его и передать управление.
  • Система может содержать несколько загрузочных разделов — администратор выбирает, какой запускать.

2. Файловые системы и тома

  • Большинство систем многопользовательские — нужна защита и совместный доступ.
  • Файлы и каталоги содержат метаданные: владелец, пользователь, права для группы и т.д.
  • Разделы накопителя используются либо для raw block I/O, либо под файловую систему.
  • Каждая ФС живёт в томе, который может состоять из одного раздела или нескольких, объединённых volume manager'ом.

3. Разделы и монтирование

  • Раздел может быть либо томом с файловой системой («cooked»), либо raw — просто последовательностью блоков.
  • Boot block указывает на загрузочный том или на набор блоков boot-загрузчика, в которых лежит код, умеющий загрузить ядро из ФС.
  • Также возможна boot management program для мультизагрузки.
  • Root partition содержит ОС; другие могут содержать иные ОС, иные ФС или быть raw.
  • Root монтируется при загрузке; остальные — автоматически или вручную.

Проверка целостности при монтировании

  • В момент монтирования проверяется консистентность метаданных.
  • Если что-то некорректно — попытка починить, повторная проверка.
  • Если всё в порядке — ФС добавляется в mount table, доступ разрешается.

4. Совместный доступ и аутентификация

  • Удалённые ФС можно реализовать просто через ftp или Web (HTTP-серверы и клиенты), либо полноценно — через client-server.
  • Mount requests и User IDs должны быть аутентифицированы, иначе любой сможет смонтировать чужую ФС.
  • Сам по себе client-server обмен не делится информацией о пользователях. Чтобы её разделять, используют распределённую информационную систему:
    • DNS — единое пространство имён;
    • Microsoft CIFS использует Active Directory, поверх которого работает Kerberos — полная служба именования и аутентификации между компьютерами сети.

5. Виртуальная файловая система (VFS)

Поддерживать несколько ФС внутри одной ОС сложно. Решение — уровневая архитектура с VFS-интерфейсом, который делает доступ к разнородным ФС прозрачным для приложений.

Системные вызовы приложения
open, read, write, close, mmap…
VFS (Virtual File System)
единый API; vnode/dentry; диспетчеризация
Конкретные реализации ФС
ext4, NTFS, ZFS, NFS, procfs, …
Блочный/сетевой ввод-вывод
  • VFS даёт один и тот же системный API для любой ФС — локальной или сетевой.
  • Разделяет общие операции ФС и детали реализации.
  • Реализует vnodes, которые хранят inode или сведения о сетевом файле.
  • Маршрутизирует вызовы в нужную реализацию ФС.

6. Реализация VFS в Linux

В Linux у VFS четыре основных объекта:

  • inode — узел индекса файла (метаданные),
  • file — открытый файл,
  • superblock — описание примонтированной ФС,
  • dentry — directory entry (элемент кэша имён каталогов).

VFS определяет набор операций, которые каждый объект должен реализовывать. У каждого объекта — указатель на таблицу функций, в которой адреса процедур, выполняющих операцию для именно этого объекта. Например:

int      open(...);    // открыть файл
int      close(...);   // закрыть файл
ssize_t  read(...);    // прочитать
ssize_t  write(...);   // записать
int      mmap(...);    // отобразить в память

Это объектно-ориентированный подход: каждая ФС поставляет свою таблицу функций, и VFS вызывает нужную через указатель.

7. Удалённые файловые системы

  • Полнофункциональная удалённая ФС включает:
    • протокол связи с удалёнными операциями;
    • трансляцию имён путей.
  • Простейшие способы — FTP или HTTP; продвинутые — client-server (NFS).

8. Семантика согласованности

Когда совместный доступ возможен, надо выбрать модель — как разруливаются одновременные обращения к одному файлу.

МодельПоведение
UNIX semanticsЗапись в открытый файл немедленно видна другим открывшим. Файловый указатель может быть разделяемым.
Session semanticsИзменения видны только сессиям, открытым после того, как файл был закрыт (AFS).
Immutable-shared-filesОбъявлённый разделяемым файл нельзя изменять — переименован, можно создать новую версию, но не править.

9. Sun NFS: концепция и монтирование

NFS — и спецификация, и реализация системы доступа к удалённым файлам через LAN/WAN. Реализована в Solaris и SunOS на рабочих станциях Sun поверх ненадёжного датаграммного протокола (UDP/IP + Ethernet).

  • Соединённые рабочие станции рассматриваются как набор независимых машин с независимыми ФС; шаринг — прозрачный.
  • Удалённый каталог монтируется поверх локального каталога — становится его поддеревом.
  • Указание удалённого каталога для mount-операции не прозрачно: нужно указать имя сервера.
  • После монтирования доступ к файлам прозрачен.
  • Любая ФС или каталог могут быть смонтированы поверх любого локального каталога, при условии прохождения проверки прав доступа.

Гетерогенность

  • NFS работает в среде разных машин, ОС и сетевых архитектур.
  • Независимость от реализации достигается через:
    • RPC-примитивы поверх
    • XDR (External Data Representation) — протокол для сериализации данных.
  • Спецификация NFS чётко разделяет:
    • mount mechanism — установление связи;
    • собственно remote file access.

Каскадное монтирование

Cascading mounts: можно монтировать ФС поверх каталога, который сам пришёл из удалённой ФС, — получается «вложенный» mount.

10. NFS Mount Protocol

  • Устанавливает начальное логическое соединение между сервером и клиентом.
  • В mount-запросе передаётся: имя удалённого каталога + имя сервера.
  • Запрос превращается в соответствующий RPC и направляется к mount-серверу на сервере.
  • Export list — список локальных ФС, которые сервер разрешает монтировать, и имена машин, которым это разрешено.
  • Если запрос соответствует export list, сервер возвращает file handle — ключ для последующих обращений:
    • идентификатор ФС;
    • номер inode внутри экспортируемой ФС, идентифицирующий смонтированный каталог.
  • Операция mount меняет только взгляд клиента; сервер о ней толком не помнит.

11. NFS Protocol

Это набор удалённых процедурных вызовов для файловых операций. Поддерживает:

  • поиск файла в каталоге;
  • чтение записей каталога;
  • работа со ссылками (links) и каталогами;
  • доступ к атрибутам файла;
  • чтение и запись файлов.
Ключевые свойства NFS:
  • Stateless servers: каждый запрос должен содержать полный набор аргументов — сервер не помнит сессии. Замечание: NFS v4 — уже stateful, сильно отличается.
  • Изменённые данные должны быть записаны на диск сервера, прежде чем результат вернётся клиенту → теряются преимущества кэширования записи.
  • Сам протокол NFS не предоставляет механизмов контроля параллелизма (concurrency control).

12. Архитектура NFS — три слоя

Интерфейс UNIX-ФС
open / read / write / close + file descriptors
VFS-слой
отличает локальные файлы от удалённых; для локальных — определяет тип ФС
NFS Service Layer
реализует NFS-протокол
  • UNIX file-system interface — открытые приложениям системные вызовы.
  • VFS layer:
    • отличает локальные файлы от удалённых;
    • для локальных запросов вызывает методы конкретной ФС;
    • для удалённых — обращается к NFS-процедурам.
  • NFS service layer — нижний слой, реализующий собственно NFS-протокол.

13. Path-name translation и кэширование

Трансляция путей

  • Путь разбивается на компоненты — для каждой пары (имя компонента, vnode каталога) выполняется отдельный NFS lookup.
  • Чтобы ускорить — на клиенте есть directory name lookup cache, который хранит vnodes для имён удалённых каталогов.

Remote operations и кэширование

  • Почти 1-к-1 соответствие обычных UNIX-вызовов и NFS RPC (кроме open/close).
  • NFS — это remote-service парадигма, но с кэшированием ради производительности.
  • File-blocks cache: при открытии файла ядро спрашивает у сервера, надо ли подгрузить или ревалидировать кэшированные атрибуты.
  • Кэшированные блоки используются только, если соответствующие атрибуты в кэше актуальны.
  • File-attribute cache: обновляется каждый раз, когда от сервера приходят новые атрибуты.
  • Клиенты не освобождают delayed-write блоки, пока сервер не подтвердит, что данные записаны на диск.
Главный вывод: внутреннее устройство файловой системы — это не одна монолитная сущность, а слоистая архитектура: разделы и тома снизу, VFS как объектно-ориентированный фасад в середине, конкретные ФС и сетевые ФС (NFS) сбоку. NFS показывает все основные компромиссы распределённой ФС: stateless-сервер ради простоты восстановления, отдельный mount-протокол с file handles, кэширование атрибутов и блоков с явной ревалидацией, и сознательный отказ от контроля параллелизма на уровне самого протокола.

1. Objectives and big picture

  • General-purpose OSes provide many file-system types, from special-purpose to general.
  • Volumes carrying file systems can be mounted into the machine's file-system space.
  • Depending on the OS, that space is either seamless (mounted FSes blend into the directory tree, as in UNIX) or distinct (each gets its own designation, as drive letters in Windows).
  • At least one FS must be bootable — it has to contain an OS. The boot loader runs first: a small program that locates the kernel in the FS, loads it, and transfers control.
  • A system may contain multiple bootable partitions — the administrator picks which to boot.

2. File systems and volumes

  • Most systems are multi-user — they need protection and file sharing.
  • Files and directories carry metadata: owner, user, group permissions, etc.
  • Storage partitions are used either for raw block I/O or for a file system.
  • Each FS lives in a volume — one partition, or several partitions coordinated by a volume manager.

3. Partitions and mounting

  • A partition is either a volume with a file system ("cooked") or raw — a plain sequence of blocks.
  • The boot block points to the boot volume or to a set of blocks holding the boot loader — enough code to find the kernel in the FS.
  • Or a boot management program for multi-OS booting.
  • The root partition holds the OS; other partitions can hold other OSes, other FSes, or be raw.
  • Root is mounted at boot; the rest mount automatically or manually.

Consistency check at mount

  • At mount time, metadata consistency is checked.
  • If something is wrong — repair and retry.
  • If OK — add to the mount table, grant access.

4. Sharing and authentication

  • Remote FSes can be set up simply via ftp or the Web (HTTP servers and clients), or fully via a client-server model.
  • Mount requests and user IDs must be authenticated — otherwise anyone could mount someone else's FS.
  • Pure client-server doesn't natively share information. To share it, a distributed information system is used:
    • DNS — a unified name space;
    • Microsoft CIFS uses Active Directory, which is built on a version of Kerberos — full naming and authentication services across the network.

5. The Virtual File System (VFS)

Supporting many FSes in one OS is hard. The solution is a layered architecture with a VFS interface that makes access to dissimilar file systems seamless to applications.

Application system calls
open, read, write, close, mmap…
VFS (Virtual File System)
unified API; vnodes/dentries; dispatch
Concrete FS implementations
ext4, NTFS, ZFS, NFS, procfs, …
Block / network I/O
  • VFS exposes the same system-call API for any FS — local or networked.
  • It separates generic FS operations from implementation details.
  • It implements vnodes, which hold inodes or remote-file details.
  • It dispatches each call to the right FS implementation routine.

6. VFS implementation in Linux

Linux's VFS has four core object types:

  • inode — file index node (metadata),
  • file — an open file,
  • superblock — a mounted FS description,
  • dentry — a directory entry (in the name cache).

VFS defines the operations each object must support. Every object has a pointer to a function table containing the addresses of routines that implement those operations on that object. For example:

int      open(...);    // open a file
int      close(...);   // close an open file
ssize_t  read(...);    // read
ssize_t  write(...);   // write
int      mmap(...);    // memory-map a file

This is the object-oriented approach: each FS provides its own function table, and VFS calls the right routine through the pointer.

7. Remote file systems

  • A full-featured remote FS includes:
    • a communication protocol with remote operations;
    • path-name translation.
  • Simpler approaches use FTP or HTTP; the polished one is client-server (NFS).

8. Consistency semantics

Once sharing is possible, you must pick a model — how concurrent accesses to one file are resolved.

ModelBehavior
UNIX semanticsA write to an open file is immediately visible to other openers. The file pointer can be shared.
Session semanticsChanges are visible only to sessions opened after the file is closed (AFS).
Immutable-shared-filesA file declared shared cannot be modified — only renamed or replaced with a new version.

9. Sun NFS: concept and mounting

NFS is both a specification and an implementation of remote-file access across LANs/WANs. It is part of Solaris and SunOS on Sun workstations, running over an unreliable datagram protocol (UDP/IP + Ethernet).

  • Interconnected workstations are seen as a set of independent machines with independent file systems; sharing is transparent.
  • A remote directory is mounted on top of a local directory — it becomes that subtree.
  • Specifying the remote directory at mount time is not transparent: you must give the server's host name.
  • After mounting, file access itself is transparent.
  • Subject to access-rights checks, any FS or any directory inside one can be mounted on top of any local directory.

Heterogeneity

  • NFS is designed for environments with different machines, OSes and network architectures.
  • Implementation independence is achieved through:
    • RPC primitives layered over
    • XDR (External Data Representation), a protocol for serializing data.
  • The NFS specification clearly separates:
    • the mount mechanism — establishing the link;
    • and the actual remote-file-access services.

Cascading mounts

You can mount a file system on top of a directory that itself came from a remote FS — yielding nested, cascading mounts.

10. NFS Mount Protocol

  • Establishes the initial logical connection between server and client.
  • The mount request carries: the remote directory name + the server name.
  • It's mapped to an RPC and forwarded to the mount server on the server machine.
  • An export list on the server specifies which local FSes it exports for mounting and which machines are allowed to mount them.
  • If the request matches the export list, the server returns a file handle — the key for further accesses:
    • a file-system identifier;
    • and an inode number identifying the mounted directory inside the exported FS.
  • The mount changes only the user's view; the server side is barely affected.

11. NFS Protocol

A set of remote procedure calls for file operations. Supports:

  • searching for a file within a directory;
  • reading directory entries;
  • manipulating links and directories;
  • accessing file attributes;
  • reading and writing files.
Key NFS properties:
  • Stateless servers: every request must carry a full set of arguments — the server doesn't remember sessions. Note: NFS v4 is stateful and very different.
  • Modified data must be committed to the server's disk before results return to the client → write-caching benefits are lost.
  • The NFS protocol itself does not provide concurrency-control mechanisms.

12. NFS architecture — three layers

UNIX file-system interface
open / read / write / close + file descriptors
VFS layer
distinguishes local vs. remote; for local — picks the FS type
NFS service layer
implements the NFS protocol
  • UNIX file-system interface — the system calls exposed to applications.
  • VFS layer:
    • distinguishes local files from remote ones;
    • for local requests it dispatches to a concrete FS's methods;
    • for remote ones it calls NFS protocol procedures.
  • NFS service layer — the bottom layer that actually speaks NFS.

13. Path-name translation and caching

Path translation

  • The path is broken into components — for each (component name, directory vnode) pair a separate NFS lookup is performed.
  • To speed this up, the client keeps a directory name lookup cache holding vnodes for remote directory names.

Remote operations and caching

  • There's a near 1-to-1 correspondence between UNIX system calls and NFS RPCs (except open/close).
  • NFS follows the remote-service paradigm but uses buffering and caching for performance.
  • File-blocks cache: when a file is opened, the kernel checks with the server whether to fetch or revalidate cached attributes.
  • Cached blocks are used only if the corresponding cached attributes are up to date.
  • File-attribute cache: updated whenever new attributes arrive from the server.
  • Clients do not free delayed-write blocks until the server confirms the data has been written to disk.
Take-away: a file system's internals form a layered stack: partitions and volumes at the bottom, VFS as an object-oriented façade in the middle, and concrete file systems (including network ones like NFS) plugged in on the side. NFS itself showcases the classic distributed-FS trade-offs: a stateless server for simple recovery, a separate mount protocol with file handles, careful attribute and block caching with explicit revalidation, and a deliberate punt on concurrency control at the protocol level.
