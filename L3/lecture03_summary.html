Processes

Процессы — резюме и пояснения

Эта лекция объясняет, что такое процесс, как ОС им управляет (состояния, PCB, планирование, переключение контекста), как процессы создаются и завершаются, и как они общаются друг с другом — через разделяемую память, сообщения, сокеты, RPC и каналы.

1. Концепция процесса

ОС выполняет разные программы: в пакетных системах — jobs, в системах разделения времени — user programs или tasks. Термины «job» и «process» часто используются взаимозаменяемо.

Процесс — это программа в исполнении. Выполнение процесса происходит последовательно.

Из чего состоит процесс

  • Text section — код программы.
  • Program counter и регистры CPU — текущая активность.
  • Stack — временные данные: параметры функций, адреса возврата, локальные переменные.
  • Data section — глобальные переменные.
  • Heap — память, выделяемая динамически во время выполнения.
Программа vs процесс: программа — пассивная сущность на диске (исполняемый файл). Процесс — активная сущность: программа становится процессом, когда исполняемый файл загружается в память (через клик мышью или ввод команды). Одна программа может породить несколько процессов — например, когда несколько пользователей запускают её одновременно.

3. Process Control Block (PCB)

PCB (он же task control block) — структура, в которой ОС хранит всю информацию о процессе:

  • Process state — running, waiting, ready и т.п.
  • Program counter — адрес следующей инструкции.
  • CPU registers — содержимое всех регистров процесса.
  • CPU scheduling information — приоритеты, указатели на очереди.
  • Memory-management information — память, выделенная процессу.
  • Accounting information — использованное CPU-время, время с момента старта, лимиты.
  • I/O status information — устройства, выделенные процессу; список открытых файлов.

Представление в Linux

Процесс описывается C-структурой task_struct:

pid_t pid;                          /* идентификатор процесса */
long state;                         /* состояние */
unsigned int time_slice;            /* инфо для планировщика */
struct task_struct *parent;         /* родительский процесс */
struct list_head children;          /* список дочерних */
struct files_struct *files;         /* открытые файлы */
struct mm_struct *mm;               /* адресное пространство */

4. Потоки (Threads)

Пока что мы считали, что процесс имеет один поток исполнения. Но можно иметь несколько program counter'ов на процесс — это потоки. Тогда:

  • Несколько мест в программе могут исполняться одновременно.
  • PCB должен хранить детали и program counter'ы каждого потока.
  • Подробно — в следующей лекции.

5. Планирование процессов

Цель планировщика — максимизировать использование CPU и быстро переключаться между процессами для разделения времени.

Очереди планирования

ОчередьЧто содержит
Job queueВсе процессы в системе.
Ready queueПроцессы в памяти, готовые к исполнению и ожидающие CPU.
Device queuesПроцессы, ожидающие конкретное устройство I/O.

Процессы мигрируют между очередями по ходу жизни.

6. Планировщики и context switch

Виды планировщиков

ПланировщикЧто делаетЧастота вызова
Short-term (CPU scheduler)Выбирает, какой процесс получит CPU следующим.Очень часто (миллисекунды) — должен быть быстрым.
Long-term (job scheduler)Выбирает, какие процессы поместить в ready queue. Контролирует степень мультипрограммирования.Редко (секунды, минуты) — может быть медленным.
Medium-termСнимает процессы из памяти на диск (swapping) и возвращает обратно, если нужно уменьшить степень многозадачности.По необходимости.

I/O-bound vs CPU-bound процессы

  • I/O-bound — больше времени тратит на I/O, много коротких CPU-всплесков.
  • CPU-bound — больше времени на вычисления, мало длинных всплесков CPU.

Long-term scheduler стремится к хорошему миксу этих типов.

Многозадачность на мобильных устройствах

  • iOS (ранние версии) — один foreground процесс + ограниченные background-процессы (короткие задачи, уведомления, аудио).
  • Android — fewer limits: foreground и background, у background есть сервис (без UI, малый расход памяти), который продолжает работать даже когда сам процесс приостановлен.

Context switch

Когда CPU переключается с одного процесса на другой, ОС должна сохранить состояние старого процесса (в его PCB) и загрузить состояние нового. Это overhead — пока идёт переключение, полезная работа не выполняется. Чем сложнее ОС и PCB, тем дольше переключение. Некоторые CPU имеют несколько наборов регистров, чтобы хранить контексты сразу.

7. Операции над процессами

Создание процесса

Родительский процесс создаёт дочерние, которые в свою очередь создают свои — образуется дерево процессов. Каждый идентифицируется через pid.

ПараметрВарианты
Совместное использование ресурсов(1) Все ресурсы общие; (2) часть; (3) ничего.
Исполнение(1) Параллельно; (2) родитель ждёт завершения детей.
Адресное пространство(1) Дочерний — дубликат родителя; (2) в дочернего загружается новая программа.
UNIX: fork() создаёт новый процесс — точную копию родителя. exec() после fork() заменяет память дочернего на новую программу.

Завершение процесса

  • exit() — процесс просит ОС удалить себя; возвращает статус родителю через wait().
  • abort() — родитель завершает дочерний (превышены ресурсы, задача больше не нужна, родитель сам выходит).
  • Cascading termination — если ОС не разрешает дочерним переживать родителя, все потомки завершаются.
  • Zombie — процесс завершился, но никто не вызвал wait().
  • Orphan — родитель завершился, не вызвав wait().

8. Межпроцессное взаимодействие (IPC)

Процессы бывают независимыми или кооперирующимися. Кооперирующиеся могут влиять друг на друга и разделять данные.

Зачем нужна кооперация

  • Совместное использование информации.
  • Ускорение вычислений.
  • Модульность.
  • Удобство.

Две модели IPC

Shared Memory

Разделяемая память

Область памяти, разделяемая процессами. Коммуникация — под контролем самих процессов, не ОС. Главная проблема — синхронизация доступа к этой памяти.

Message Passing

Передача сообщений

Процессы общаются через send(message) / receive(message), не разделяя переменных. ОС берёт на себя пересылку. Размер сообщений — фиксированный или переменный.

Producer-Consumer Problem

Классическая парадигма кооперации: процесс-производитель создаёт информацию, процесс-потребитель её потребляет.

  • Unbounded buffer — без практического ограничения размера.
  • Bounded buffer — фиксированный размер.

9. Передача сообщений

Если процессы P и Q хотят общаться, им нужно: (1) установить link, (2) обмениваться сообщениями через send/receive. Реализация может быть физической (shared memory, аппаратная шина, сеть) или логической.

Прямая vs косвенная коммуникация

Прямая (Direct)Косвенная (Indirect)
АдресацияЯвно по имени процесса: send(P, msg)Через mailbox / port: send(A, msg)
Установка linkАвтоматически.Только если процессы разделяют общий mailbox.
СвязьРовно одна пара процессов на link.Один link может быть с несколькими процессами.
НаправлениеОбычно двунаправленная.Одно- или двунаправленная.

Mailbox sharing. Если P1, P2, P3 разделяют mailbox A, P1 отправляет, а P2 и P3 получают — кто получит сообщение? Варианты: разрешить link только двум процессам; разрешить receive только одному за раз; пусть система сама выбирает (и сообщает sender'у).

Синхронизация: blocking vs non-blocking

ТипСемантика
Blocking sendSender блокирован до получения сообщения.
Blocking receiveReceiver блокирован, пока сообщение не доступно.
Non-blocking sendSender отправляет и продолжает работу.
Non-blocking receiveПолучатель получает либо сообщение, либо null.

Blocking = синхронно, non-blocking = асинхронно. Если оба blocking — это rendezvous.

Буферизация

  • Zero capacity — сообщения не очередь не ставятся; sender ждёт receiver (rendezvous).
  • Bounded capacity — очередь конечная (n сообщений); sender ждёт при переполнении.
  • Unbounded capacity — бесконечная очередь; sender никогда не ждёт.

Пример: Windows LPC

Windows реализует IPC через Local Procedure Call (LPC) — только между процессами на одной машине. Используются порты (как mailboxes):
  1. Клиент открывает handle к connection port подсистемы.
  2. Клиент отправляет запрос на соединение.
  3. Сервер создаёт два приватных порта и возвращает handle одного клиенту.
  4. Клиент и сервер обмениваются сообщениями/callback'ами через эти handles.

10. Коммуникации клиент-сервер

В сетевых системах используются четыре основных механизма:

Sockets

Сокеты

Socket = endpoint коммуникации, IP-адрес + порт. Например, 161.25.19.8:1625. Коммуникация — между парой сокетов.

  • Порты ниже 1024 — well-known, стандартные службы.
  • 127.0.0.1 — loopback, обращение к локальной машине.
RPC

Remote Procedure Calls

Абстрагирует вызов процедур между процессами в сети. Использует порты для различения сервисов.

  • Stubs — прокси на стороне клиента: находит сервер и marshals параметры.
  • Server-side stub принимает сообщение, unmarshals параметры, выполняет процедуру.
  • XDL (External Data Representation) — общий формат данных для разных архитектур (big/little-endian).
  • В Windows stub генерируется из MIDL.
  • Семантика — exactly once вместо at-most-once.
  • ОС предоставляет rendezvous (matchmaker) для соединения клиента и сервера.
Pipes

Каналы (Pipes)

Канал — conduit для двух процессов. Главные вопросы: одно- или двунаправленный; нужны ли parent-child отношения; работает ли через сеть.

RMI

Remote Method Invocation

Java-механизм; концептуально похож на RPC, но для объектов.

Ordinary vs Named Pipes

OrdinaryNamed
НаправлениеОднонаправленный (producer→consumer).Двунаправленный.
Parent-childОбязательно.Не нужно.
Кол-во процессовТолько создавший процесс (и его потомки).Несколько процессов могут использовать.
Windows-аналогAnonymous pipes.Named pipes (есть и в UNIX, и в Windows).

Главные выводы

  • Процесс = программа в исполнении + код, стек, данные, heap, регистры.
  • ОС хранит состояние процесса в PCB и переключает контексты.
  • Состояния: new → ready ⇄ running → terminated; running → waiting → ready.
  • Три уровня планирования: short-term (CPU), long-term (admission), medium-term (swap).
  • Процессы образуют дерево; UNIX использует fork() + exec().
  • Два IPC: shared memory (быстро, но требует синхронизации) и message passing (под управлением ОС).
  • Сетевые IPC: Sockets, RPC, Pipes, RMI.

1. Process Concept

An OS runs many kinds of programs: batch systems call them jobs, time-shared systems call them user programs or tasks. The terms "job" and "process" are often used interchangeably.

A process is a program in execution. Execution must progress in sequential fashion.

Parts of a process

  • Text section — the program code.
  • Program counter and CPU registers — current activity.
  • Stack — temporary data: function parameters, return addresses, local variables.
  • Data section — global variables.
  • Heap — memory dynamically allocated at runtime.
Program vs process: a program is a passive entity on disk (executable file). A process is active — a program becomes a process when its executable is loaded into memory (via a mouse click, command-line entry, etc.). One program can spawn several processes — for instance, when multiple users run it concurrently.

3. Process Control Block (PCB)

The PCB (also called task control block) holds everything the OS needs to know about a process:

  • Process state — running, waiting, ready, etc.
  • Program counter — address of the next instruction.
  • CPU registers — contents of all process-centric registers.
  • CPU scheduling information — priorities, queue pointers.
  • Memory-management information — memory allocated to the process.
  • Accounting information — CPU time used, clock time elapsed, time limits.
  • I/O status information — devices allocated, list of open files.

Linux representation

A process is described by the C structure task_struct:

pid_t pid;                          /* process identifier */
long state;                         /* state of the process */
unsigned int time_slice;            /* scheduling information */
struct task_struct *parent;         /* parent process */
struct list_head children;          /* this process's children */
struct files_struct *files;         /* list of open files */
struct mm_struct *mm;               /* address space */

4. Threads

So far a process has had a single thread of execution. With multiple program counters per process, several locations can execute at once — these are threads. Then:

  • Multiple code locations execute concurrently within one process.
  • The PCB stores per-thread details and program counters.
  • Covered in detail in the next chapter.

5. Process Scheduling

The scheduler aims to maximize CPU use and switch processes quickly for time sharing.

Scheduling queues

QueueContents
Job queueAll processes in the system.
Ready queueProcesses residing in main memory, ready and waiting to execute.
Device queuesProcesses waiting for a particular I/O device.

Processes migrate between queues over their lifetime.

6. Schedulers and Context Switch

Types of schedulers

SchedulerRoleFrequency
Short-term (CPU scheduler)Selects which process runs next; allocates CPU.Very often (ms) — must be fast.
Long-term (job scheduler)Selects which processes enter the ready queue. Controls the degree of multiprogramming.Rarely (seconds, minutes) — may be slow.
Medium-termRemoves processes from memory to disk (swapping) and brings them back, to reduce multiprogramming when needed.As needed.

I/O-bound vs CPU-bound processes

  • I/O-bound — spends more time on I/O than computation; many short CPU bursts.
  • CPU-bound — spends more time computing; few very long CPU bursts.

The long-term scheduler aims for a good mix of the two.

Multitasking on mobile systems

  • iOS (early versions) — a single foreground process + limited background processes (short tasks, notifications, audio).
  • Android — fewer limits: both foreground and background. Background processes use a service (no UI, low memory) that can keep running even when the parent process is suspended.

Context switch

When the CPU switches to another process, the OS must save the old process's state (into its PCB) and load the new one's. This is pure overhead — no useful work happens during the switch. The richer the OS and PCB, the longer the switch takes. Some CPUs provide multiple register sets so multiple contexts can be loaded at once.

7. Operations on Processes

Process creation

A parent process creates children, which in turn create more — forming a tree of processes. Each is identified by a pid.

DimensionOptions
Resource sharing(1) Parent and children share all; (2) children share a subset; (3) no sharing.
Execution(1) Concurrent; (2) parent waits for children to terminate.
Address space(1) Child is a duplicate of parent; (2) child loads a new program.
UNIX: fork() creates a new process — an exact copy of the parent. exec() after fork() replaces the child's memory with a new program.

Process termination

  • exit() — the process asks the OS to delete it; returns status to the parent via wait().
  • abort() — parent kills a child (child exceeded resources, task no longer needed, parent exiting).
  • Cascading termination — if the OS doesn't allow children to outlive their parent, all descendants are terminated.
  • Zombie — process has terminated but no one called wait().
  • Orphan — parent terminated without calling wait().

8. Interprocess Communication (IPC)

Processes are either independent or cooperating. Cooperating processes can affect each other and share data.

Why cooperate?

  • Information sharing.
  • Computation speedup.
  • Modularity.
  • Convenience.

Two IPC models

Shared Memory

Shared memory

A memory region shared between processes. Communication is controlled by the processes themselves, not the OS. The big challenge is synchronization of access to that region.

Message Passing

Message passing

Processes use send(message) / receive(message) without sharing variables. The OS does the delivery. Message size: fixed or variable.

Producer-Consumer Problem

A classic cooperation pattern: a producer process produces information that a consumer process consumes.

  • Unbounded buffer — no practical limit on size.
  • Bounded buffer — fixed buffer size.

9. Message Passing

For processes P and Q to communicate, they must (1) establish a link and (2) exchange messages via send/receive. Links can be physical (shared memory, hardware bus, network) or logical.

Direct vs indirect communication

DirectIndirect
AddressingBy process name: send(P, msg)Through a mailbox / port: send(A, msg)
Link setupAutomatic.Only if processes share a common mailbox.
AssociationExactly one pair of processes per link.A link may be shared by many processes.
DirectionUsually bi-directional.Uni- or bi-directional.

Mailbox sharing. If P1, P2, P3 share mailbox A, and P1 sends while P2 and P3 both receive — who gets the message? Options: allow at most two processes per link; allow only one receive at a time; let the system pick arbitrarily and notify the sender.

Synchronization: blocking vs non-blocking

TypeSemantics
Blocking sendSender blocks until the message is received.
Blocking receiveReceiver blocks until a message is available.
Non-blocking sendSender sends and continues.
Non-blocking receiveReceiver gets either a valid message or a null.

Blocking = synchronous, non-blocking = asynchronous. If both send and receive are blocking, the pattern is a rendezvous.

Buffering

  • Zero capacity — no messages queued; sender must wait for the receiver (rendezvous).
  • Bounded capacity — queue of finite length n; sender waits if full.
  • Unbounded capacity — infinite queue; sender never waits.

Example: Windows LPC

Windows uses an advanced Local Procedure Call (LPC) facility — only between processes on the same machine. It uses ports (like mailboxes):
  1. Client opens a handle to the subsystem's connection port.
  2. Client sends a connection request.
  3. Server creates two private communication ports and returns the handle to one.
  4. Client and server exchange messages or callbacks through those handles.

10. Client-Server Communication

In networked systems, four mechanisms dominate:

Sockets

Sockets

A socket is an endpoint of communication — IP address + port. Example: 161.25.19.8:1625. Communication is between a pair of sockets.

  • Ports below 1024 are well-known, reserved for standard services.
  • 127.0.0.1 is the loopback address — the local machine.
RPC

Remote Procedure Calls

Abstracts procedure calls across networked processes. Uses ports to distinguish services.

  • Stubs — client-side proxy: locates the server and marshals the parameters.
  • Server-side stub receives the message, unmarshals the parameters, runs the procedure.
  • XDL (External Data Representation) handles cross-architecture differences (big- vs little-endian).
  • On Windows, stubs are generated from MIDL.
  • Semantics: exactly-once instead of at-most-once.
  • OS provides a rendezvous (matchmaker) service to wire up client and server.
Pipes

Pipes

A pipe is a conduit between two processes. Key questions: uni- or bi-directional; required parent-child relationship; does it work across the network.

RMI

Remote Method Invocation

Java's mechanism; conceptually like RPC but for objects.

Ordinary vs Named Pipes

OrdinaryNamed
DirectionUnidirectional (producer→consumer).Bidirectional.
Parent-childRequired.Not required.
Number of processesOnly the creating process (and its children).Many processes can use it.
Windows analogueAnonymous pipes.Named pipes (exist on both UNIX and Windows).

Key Takeaways

  • A process = program in execution + code, stack, data, heap, registers.
  • The OS keeps process state in a PCB and switches contexts between processes.
  • States: new → ready ⇄ running → terminated; running → waiting → ready.
  • Three scheduling layers: short-term (CPU), long-term (admission), medium-term (swap).
  • Processes form a tree; UNIX uses fork() + exec().
  • Two IPC models: shared memory (fast, needs synchronization) and message passing (OS-managed).
  • Networked IPC: Sockets, RPC, Pipes, RMI.
