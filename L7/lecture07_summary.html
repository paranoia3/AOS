File-System Interface

Интерфейс файловой системы — резюме и пояснения

Эта лекция описывает интерфейс файловой системы: что такое файл как абстрактный тип данных, какие у него атрибуты и операции, как файлы организованы в каталогах (single-level → two-level → tree → acyclic-graph → general graph), как работают методы доступа (sequential, direct, index), как монтируются файловые системы, как делятся файлы между пользователями и по сети (NFS, CIFS, LDAP, DNS), и как обеспечивается защита через access-lists и группы прав rwx.

1. Концепция файла

Файл — это абстрактный тип данных, определяемый и реализуемый ОС. Он представляет собой последовательность логических записей (logical records) в непрерывном логическом адресном пространстве.

Логическая запись может быть:

  • байтом,
  • строкой (фиксированной или переменной длины),
  • более сложным элементом данных.

Поддержка тех или иных типов записей может быть в ОС или оставлена прикладной программе.

Задача ОС: отобразить логическую концепцию файла на физическое устройство хранения (HDD, NVM). Так как физический размер записи устройства может отличаться от логического, ОС или приложение упаковывают логические записи в физические.

Типы содержимого

  • Данные: числовые, символьные, бинарные;
  • Программы.

Содержание определяется создателем файла: примеры — text file, source file, executable file.

Информация о файлах хранится в структуре каталога на диске. Существуют расширенные атрибуты — например, контрольная сумма файла.

3. Операции над файлами и таблица открытых файлов

Файл — это ADT. Базовые операции:

  • Create — создать;
  • Write — записать по указателю записи;
  • Read — прочитать по указателю чтения;
  • Reposition (seek) — переместиться внутри файла;
  • Delete — удалить;
  • Truncate — обрезать (содержимое выкинуть, атрибуты оставить);
  • Open(Fi) — найти запись Fi в каталоге на диске и подгрузить в память;
  • Close(Fi) — записать запись Fi из памяти обратно в каталог.

Open File Table

Чтобы управлять открытыми файлами, ОС хранит:

  • Open-file table — список открытых файлов;
  • File pointer — указатель последнего чтения/записи, свой у каждого процесса;
  • File-open count — счётчик, сколько процессов открыли файл (когда падает до 0 — запись удаляется из таблицы);
  • Disk location — кэш сведений о месте файла на диске;
  • Access rights — режим доступа, в каком файл открыт каждым процессом.

4. Блокировки файлов (File Locking)

Поддерживается некоторыми ОС/ФС, аналог reader-writer locks.

  • Shared lock — аналог reader lock: несколько процессов могут держать одновременно.
  • Exclusive lock — аналог writer lock: только один.

Типы:

  • Mandatory — ОС запрещает доступ по тому, какие блокировки заняты или запрашиваются.
  • Advisory — ОС лишь сообщает статус блокировок; процессы сами решают, что делать.

Пример: Java API.

RandomAccessFile raf = new RandomAccessFile("file.txt", "rw");
FileChannel ch = raf.getChannel();

// эксклюзивная блокировка первой половины файла
FileLock exclusiveLock = ch.lock(0, raf.length()/2, EXCLUSIVE);
/* модифицируем данные… */
exclusiveLock.release();

// shared-блокировка второй половины
FileLock sharedLock = ch.lock(raf.length()/2+1, raf.length(), SHARED);
/* читаем данные… */
sharedLock.release();

5. Типы файлов и структура

Тип файла обычно кодируется в имени (расширении): .txt, .exe, .c, .html и т.д. ОС или приложение по этому расширению понимает, как обрабатывать содержимое.

Возможные структуры файла

  • None — просто последовательность байт/слов.
  • Simple record structure — строки фиксированной или переменной длины.
  • Complex structures — форматированный документ, relocatable load file.

Сложные структуры можно симулировать поверх простой последовательности байт, вставляя управляющие символы. Решение «кто определяет структуру» — ОС или сама программа.

6. Методы доступа

Sequential Access

  • read next — прочитать следующую запись;
  • write next — записать следующую;
  • reset — вернуться в начало;
  • нельзя читать после последней записи (только переписать).

Direct Access

Файл — это набор logical records фиксированной длины:

  • read n, write n, rewrite n, position to n,
  • где n — relative block number (относительный номер блока).

Относительные номера дают ОС свободу размещать блоки файла где угодно физически.

Sequential-доступ можно симулировать поверх direct-access ФС с помощью текущего указателя.

Index Access

  • Поверх базовых методов строится индекс файла.
  • Индекс держат в памяти — быстро находим расположение данных (пример: UPC-код + запись об этом товаре).
  • Если индекс слишком большой — строится индекс индекса.
  • Классика: IBM ISAM (Indexed Sequential-Access Method) — маленький master-index, указывающий на блоки secondary-index; файл отсортирован по ключу. Всё ведёт ОС.
  • VMS предоставляет index- и relative-файлы.

7. Структура диска и тома

  • Диск можно разбить на partitions (= minidisks, slices).
  • Диски/разделы могут быть защищены RAID.
  • Раздел можно использовать raw (без ФС) или отформатировать под ФС.
  • Сущность, содержащая файловую систему, называется томом (volume). Том ведёт device directory или volume table of contents с метаданными ФС.

8. Типы файловых систем

Обычно говорят о general-purpose ФС, но в одной системе их часто несколько — общего и специального назначения. Например, в Solaris:

  • tmpfs — ФС в памяти для быстрого временного I/O;
  • objfs — доступ к памяти ядра для отладки символов;
  • ctfs — contract file system для управления демонами;
  • lofs — loopback FS, позволяет монтировать ФС в место другой;
  • procfs — интерфейс к структурам процессов в ядре;
  • ufs, zfs — общего назначения.

9. Структуры каталогов

Каталог — это набор узлов, содержащих информацию о всех файлах. И каталог, и сами файлы хранятся на диске.

Операции над каталогом

  • поиск файла, создание/удаление файла, листинг каталога, переименование, обход всей ФС.

Цели организации

  • Efficiency — быстрый поиск.
  • Naming — удобство для пользователя: два разных файла у разных пользователей могут иметь одно имя; один файл — несколько имён.
  • Grouping — логическая группировка (все Java-программы, все игры…).

Single-Level Directory

  • Один каталог на всех пользователей.
  • Проблемы: конфликты имён, нет группировки.

Two-Level Directory

  • Свой каталог для каждого пользователя.
  • Появляется path name; разные пользователи могут иметь файлы с одинаковыми именами.
  • Эффективный поиск; нет группировки.

Tree-Structured Directories

  • Естественное обобщение: каждый каталог может содержать подкаталоги.
  • Эффективный поиск + полноценная группировка.
  • Current directory (working directory): cd /spell/mail/prog.
  • Absolute и relative path names.
  • Новый файл/подкаталог создаётся в текущей директории: mkdir <dir>, rm <file>.
  • Удаление каталога удаляет всё поддерево.
/ ├── spell/ │ ├── mail/ │ │ ├── prog/ │ │ └── count │ └── ... ├── home/ │ ├── alice/ │ └── bob/ └── etc/

Acyclic-Graph Directories

  • Позволяют разделять подкаталоги и файлы между ветвями.
  • Aliasing — у одного файла несколько имён (links).
  • Усложняет поиск и удаление.

Проблема: если dict удалит list, остаются dangling pointers. Решения:

  • Backpointers — обратные ссылки, чтобы удалить все указатели; проблема — записи переменной длины.
  • Backpointers через daisy-chain.
  • Entry-hold-count — счётчик ссылок, файл реально удаляется при достижении нуля.
  • Новый тип записи каталога — link (просто другой указатель на существующий файл); «разрешить link» = пройти по указателю.

General Graph Directory

  • Полная свобода в разделении, но возможны циклы.
  • Иногда требуется garbage collection для возврата неиспользованного места.

Как избежать циклов?

  • Разрешать ссылки только на файлы, не на каталоги;
  • Garbage collection;
  • При добавлении новой ссылки — алгоритм обнаружения цикла.

10. Монтирование файловой системы

  • Файловая система должна быть смонтирована, прежде чем к ней можно обращаться.
  • Несмонтированная ФС подключается в mount point — точку в существующем дереве каталогов.

11. Совместный доступ к файлам

  • На многопользовательских системах общий доступ к файлам желателен.
  • Регулируется схемой защиты.
  • В распределённых системах файлы могут шариться по сети. Стандарт UNIX — NFS.

Идентификация в многопользовательской системе

  • User ID — идентификатор пользователя, по нему дают права.
  • Group ID — членство в группе для групповых прав.
  • У каждого файла/каталога есть owner и group.

12. Удалённые ФС и семантика согласованности

Remote File Systems

  • Доступ к файлам между системами по сети.
  • Способы: вручную (FTP), автоматически и прозрачно (распределённые ФС), полуавтоматически (Web).
  • Client-server: клиент монтирует удалённую ФС, сервер может обслуживать многих клиентов.
  • Идентификация клиента/пользователя — небезопасна или сложна.
  • NFS — стандартный UNIX-протокол, CIFS — Windows.
  • Обычные системные файловые вызовы переводятся в удалённые.
  • Distributed Information Systems (LDAP, DNS, NIS, Active Directory) дают единообразный доступ к информации, нужной для удалённой работы.

Режимы отказов

  • У всех ФС бывают сбои — например, повреждение каталоговых структур или другой metadata.
  • У remote ФС добавляются новые: сбой сети, сбой сервера.
  • Восстановление может требовать хранения состояния каждого удалённого запроса.
  • Stateless-протоколы (например, NFS v3) содержат всю информацию в каждом запросе — простое восстановление, но меньше безопасности.

Consistency Semantics

Задаёт, как несколько пользователей видят одновременные изменения общего файла. Аналог процессной синхронизации, но обычно проще из-за латентности I/O и сети.

СистемаСемантика
UFS (Unix)Записи в открытый файл немедленно видны другим открывшим. Указатель файла может разделяться, позволяя нескольким пользователям одновременно читать и писать.
AFS (Andrew)Session semantics: записи видны только тем сессиям, которые открыли файл после того, как он был закрыт.

13. Защита и права доступа

Владелец/создатель файла должен иметь возможность контролировать, что можно делать и кому.

Типы доступа

  • Read, Write, Execute, Append, Delete, List.

Access Lists и группы (UNIX/Linux)

Режимы доступа: R, W, X. Три класса пользователей:

КлассБитыВосьмеричное
ownerrwx = 1117
grouprw- = 1106
public--x = 0011

Чтобы выдать групповой доступ:

  1. Попросить администратора создать группу с уникальным именем (скажем, G) и добавить в неё пользователей.
  2. Для конкретного файла или подкаталога задать соответствующий режим.
  3. Привязать группу к файлу: chgrp G game.

Windows ACL

Windows 7 предоставляет полноценные Access-Control Lists: к каждому файлу/каталогу можно прикрепить список с правами для конкретных пользователей и групп — гораздо тоньше, чем модель owner/group/public.

Главный вывод: файл — это абстракция ОС с богатым набором атрибутов и операций. Каталоги нужны как для организации (одноуровневые → деревья → графы), так и для защиты и удобства именования. Методы доступа (sequential, direct, index) дают разные компромиссы между простотой и производительностью. Совместный доступ требует моделей пользователей/групп и (в распределённой среде) — отдельных протоколов вроде NFS/CIFS со своими семантиками согласованности. Защита реализуется через права rwx или полноценные ACL.

1. File concept

A file is an abstract data type defined and implemented by the OS. It is a sequence of logical records in a contiguous logical address space.

A logical record can be:

  • a byte,
  • a line (fixed or variable length),
  • a more complex data item.

The OS may explicitly support certain record types, or leave it to the application.

A central OS task: map the logical file onto a physical storage device (HDD, NVM). Since physical record sizes may differ from logical ones, the OS or the application must pack logical records into physical ones.

Content types

  • Data: numeric, character, binary;
  • Programs.

Content is defined by the file's creator: text file, source file, executable file, etc.

2. File attributes

AttributePurpose
NameThe only attribute in human-readable form.
IdentifierA unique tag (number) identifying the file inside the FS.
TypeFor systems that support multiple file types.
LocationPointer to the file's location on the device.
SizeCurrent file size.
ProtectionControls who can read, write, execute.
Time, date, userFor protection, security, usage monitoring.

File metadata lives in the directory structure on disk. Extended attributes (e.g., file checksum) are common.

3. File operations and the open-file table

A file is an ADT. Core operations:

  • Create;
  • Write at the write pointer;
  • Read at the read pointer;
  • Reposition (seek) inside the file;
  • Delete;
  • Truncate — drop content but keep attributes;
  • Open(Fi) — find the directory entry Fi on disk and pull it into memory;
  • Close(Fi) — write the memory copy of Fi back to the directory structure.

Open-file table

To manage open files the OS tracks:

  • Open-file table — list of currently open files;
  • File pointer — last read/write position, per process;
  • File-open count — how many processes have it open (drops to 0 → entry removed);
  • Disk location — cached lookup info;
  • Access rights — per-process access mode.

4. File locking

Provided by some OSes and FSes, analogous to reader-writer locks.

  • Shared lock ≈ reader lock: multiple processes can hold it.
  • Exclusive lock ≈ writer lock: only one process.

Types:

  • Mandatory — the OS denies access based on which locks are held vs. requested.
  • Advisory — the OS only reports lock status; processes decide what to do.

Java API example:

RandomAccessFile raf = new RandomAccessFile("file.txt", "rw");
FileChannel ch = raf.getChannel();

// exclusive lock on the first half
FileLock exclusiveLock = ch.lock(0, raf.length()/2, EXCLUSIVE);
/* modify the data… */
exclusiveLock.release();

// shared lock on the second half
FileLock sharedLock = ch.lock(raf.length()/2+1, raf.length(), SHARED);
/* read the data… */
sharedLock.release();

5. File types and structure

The file's type is usually encoded in the name extension: .txt, .exe, .c, .html, etc. The OS or application uses this to know how to handle the contents.

Possible structures

  • None — a sequence of bytes/words.
  • Simple record structure — fixed or variable-length lines.
  • Complex structures — formatted document, relocatable load file.

Complex structures can be simulated on top of a flat byte stream by inserting control characters. Who decides: the OS or the program.

6. Access methods

Sequential Access

  • read next;
  • write next;
  • reset — back to the beginning;
  • no read after the last write (only rewrite).

Direct Access

The file is a set of fixed-length logical records:

  • read n, write n, rewrite n, position to n;
  • where n is the relative block number.

Relative block numbers let the OS place file blocks anywhere physically.

Sequential access can be simulated on top of a direct-access FS using a current pointer.

Index access

  • Built on top of the base methods, an index for the file is created.
  • The index is kept in memory for fast lookup (think UPC code + data record).
  • If the index is too big, an index of the index is built.
  • Classic: IBM ISAM (Indexed Sequential-Access Method) — a small master index points to secondary-index blocks; the file is sorted on a key. All managed by the OS.
  • VMS provides index and relative files.

7. Disk structure and volumes

  • A disk can be split into partitions (= minidisks, slices).
  • Disks or partitions can be RAID-protected.
  • A partition can be used raw (no FS) or formatted with one.
  • An entity holding a file system is a volume. It maintains a device directory or volume table of contents with FS metadata.

8. Types of file systems

We mostly talk about general-purpose FSes, but real systems typically run several — some general, some special. Solaris, for example:

  • tmpfs — RAM-backed, volatile FS for fast temporary I/O;
  • objfs — interface into kernel memory for debug symbols;
  • ctfs — contract file system for daemon management;
  • lofs — loopback FS that mounts one FS in place of another;
  • procfs — kernel interface to process structures;
  • ufs, zfs — general-purpose.

9. Directory structures

A directory is a collection of nodes holding info about all files. Both the directory and the files live on disk.

Operations on a directory

  • search for a file, create/delete, list, rename, traverse the FS.

Design goals

  • Efficiency — find a file fast.
  • Naming — user-friendly: two users may have files with the same name; one file may have multiple names.
  • Grouping — logical grouping (all Java programs, all games…).

Single-Level Directory

  • One directory for everyone.
  • Problems: name collisions, no grouping.

Two-Level Directory

  • A separate directory per user.
  • Introduces path names; different users can have files with identical names.
  • Efficient search; no grouping.

Tree-Structured Directories

  • The natural generalization: each directory may hold subdirectories.
  • Efficient search + full grouping.
  • Current (working) directory: cd /spell/mail/prog.
  • Absolute and relative path names.
  • New files/subdirectories are created in the current directory: mkdir <dir>, rm <file>.
  • Deleting a directory deletes the entire subtree.
/ ├── spell/ │ ├── mail/ │ │ ├── prog/ │ │ └── count │ └── ... ├── home/ │ ├── alice/ │ └── bob/ └── etc/

Acyclic-Graph Directories

  • Allow sharing of subdirectories and files between branches.
  • Aliasing — one file with multiple names (links).
  • Complicates search and deletion.

Problem: if dict deletes list, dangling pointers remain. Solutions:

  • Back-pointers — to delete all aliases; variable-size records are awkward.
  • Daisy-chain back-pointers.
  • Entry-hold-count — a reference count; the file is removed only at zero.
  • A new entry type — link (just another pointer to an existing file); "resolve the link" = follow the pointer.

General Graph Directory

  • Full sharing freedom, but cycles become possible.
  • Sometimes requires garbage collection to reclaim unused space.

Avoiding cycles:

  • Only allow links to files, not to subdirectories;
  • Garbage collection;
  • Run a cycle-detection algorithm whenever a new link is added.

10. File-system mounting

  • A file system must be mounted before it can be accessed.
  • An unmounted FS is attached at a mount point — a node in the existing directory tree.

11. File sharing

  • Sharing on multi-user systems is desirable.
  • Governed by a protection scheme.
  • Across a network, files may be shared by distributed FSes. The UNIX standard is NFS.

Multi-user identification

  • User ID — identifies a user; permissions are per-user.
  • Group ID — group membership for group rights.
  • Every file/directory has an owner and a group.

12. Remote FS and consistency semantics

Remote File Systems

  • Network-based access to files across systems.
  • Modes: manually (FTP), automatically and seamlessly (distributed FSes), semi-automatically (the Web).
  • Client-server: the client mounts the remote FS; one server can serve many clients.
  • Identification of client/user is insecure or complicated.
  • NFS — the standard UNIX protocol; CIFS — the standard Windows one.
  • Ordinary file calls are translated into remote calls.
  • Distributed information systems (LDAP, DNS, NIS, Active Directory) unify access to information needed for remote computing.

Failure modes

  • All FSes can fail — e.g., corruption of directory structures or other metadata.
  • Remote FSes add network failure and server failure.
  • Recovery may require keeping state about every remote request.
  • Stateless protocols (such as NFS v3) carry all info in each request — easy recovery, weaker security.

Consistency Semantics

Specifies how concurrent users see each other's writes to a shared file. Similar to process synchronization but usually simpler due to I/O and network latency.

SystemSemantics
UFS (Unix)Writes to an open file are immediately visible to others who have the file open. The file pointer can be shared so several users read and write concurrently.
AFS (Andrew)Session semantics: writes become visible only to sessions opened after the file is closed.

13. Protection and permissions

The owner/creator of a file should be able to control what can be done and by whom.

Access types

  • Read, Write, Execute, Append, Delete, List.

Access lists and groups (UNIX/Linux)

Modes: R, W, X. Three user classes:

ClassBitsOctal
ownerrwx = 1117
grouprw- = 1106
public--x = 0011

To grant group access:

  1. Ask the admin to create a uniquely-named group (say G) and add users.
  2. Set the desired mode on the file or subdirectory.
  3. Attach the group to the file: chgrp G game.

Windows ACLs

Windows 7 provides full Access-Control Lists: each file/directory has a list assigning rights to specific users and groups — much finer than the owner/group/public model.

Take-away: a file is an OS abstraction with a rich set of attributes and operations. Directories organize files and support protection and naming — they grow from single-level into trees and ultimately graphs. Access methods (sequential, direct, indexed) trade simplicity for performance. Sharing requires a user/group model and, in distributed settings, protocols like NFS/CIFS with their own consistency semantics. Protection is enforced via rwx bits or full ACLs.
