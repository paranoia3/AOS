CPU Scheduling

Планирование ЦП — резюме и пояснения

Эта лекция объясняет, как ОС выбирает, какой процесс/поток запустить на ЦП: цикл CPU–I/O burst, роль короткого планировщика и диспетчера, критерии качества (utilization, throughput, turnaround, waiting, response), классические алгоритмы (FCFS, SJF, Priority, Round Robin, многоуровневые очереди), планирование потоков и в SMP/NUMA, real-time (Rate Monotonic, EDF), реальные планировщики Linux CFS / Windows / Solaris и способы оценки алгоритмов (детерминированное моделирование, очереди и формула Литтла, симуляция, реализация).

1. Базовые концепции и CPU–I/O burst

  • Максимальная загрузка ЦП достигается за счёт мультипрограммирования.
  • Выполнение процесса представляет собой цикл: CPU burst → I/O wait → CPU burst → I/O wait → …
  • За CPU burst следует I/O burst; для планирования главное — распределение длительностей CPU burst.
Наблюдение: гистограмма длительностей CPU burst обычно сильно скошена в сторону коротких бёрстов — короткие встречаются гораздо чаще длинных. Это и есть теоретическая основа эффективности SJF.

2. Короткий планировщик и диспетчер

Short-term scheduler (короткий планировщик) выбирает из ready-очереди процесс, которому отдадут ЦП. Очередь может быть упорядочена по-разному.

Решение о переключении принимается, когда процесс:

  1. переходит из running в waiting (например, ждёт I/O);
  2. переходит из running в ready (например, по прерыванию);
  3. переходит из waiting в ready (завершилось I/O);
  4. завершается.

Планирование в случаях 1 и 4 — non-preemptive (без вытеснения). Все остальные — preemptive (с вытеснением). При вытеснении надо учитывать: доступ к разделяемым данным, вытеснение в kernel mode, прерывания во время критичных действий ОС.

Диспетчер

Диспетчер передаёт управление ЦП выбранному процессу. Это включает:

  • переключение контекста,
  • переход в user mode,
  • переход в нужное место пользовательской программы для её возобновления.

Dispatch latency — время, которое диспетчер тратит на остановку одного процесса и запуск другого.

3. Критерии планирования

CPU utilization
Держать ЦП максимально занятым.
Throughput
Число завершённых процессов в единицу времени.
Turnaround time
Полное время выполнения процесса.
Waiting time
Сколько процесс провёл в ready-очереди.
Response time
Время от подачи запроса до первой реакции (не до полного ответа). Важно в интерактивных системах.

Цели оптимизации: максимум utilization и throughput; минимум turnaround, waiting и response time.

4. First-Come, First-Served (FCFS)

Простейший алгоритм: процесс, пришедший первым, обслуживается первым.

Пример 1. Порядок прихода: P1(24), P2(3), P3(3).

P10–24
P224–27
P327–30

Waiting time: P1=0, P2=24, P3=27. Среднее: (0+24+27)/3 = 17.

Пример 2. Порядок прихода: P2, P3, P1.

P20–3
P33–6
P16–30

Среднее ожидание: (6+0+3)/3 = 3 — намного лучше.

Convoy effect: короткие процессы стоят в очереди за одним длинным — все ждут. Особенно плох FCFS, когда есть один CPU-bound и много I/O-bound процессов.

5. Shortest-Job-First (SJF) и SRTF

  • С каждым процессом связана длина его следующего CPU burst.
  • Выбирается процесс с минимальной ожидаемой длиной.
  • SJF оптимален: даёт минимальное среднее waiting time для данного набора процессов.
  • Проблема: длина следующего CPU burst неизвестна — её приходится оценивать.

Оценка длины следующего CPU burst

Используется экспоненциальное усреднение прошлых длительностей:

τn+1 = α · tn + (1 − α) · τn

где tn — длина последнего реального burst, τn — предыдущая оценка, обычно α = ½.

  • α = 0: τn+1 = τn — недавняя история не учитывается.
  • α = 1: τn+1 = tn — учитывается только последний burst.
  • Разложение: τn+1 = α·tn + (1−α)·α·tn−1 + … — вклад каждого предыдущего значения экспоненциально убывает.

Shortest-Remaining-Time-First (SRTF)

Preemptive-версия SJF: если приходит новый процесс с меньшим оставшимся CPU burst, чем у текущего, — текущий вытесняется. Среднее waiting time ещё меньше.

6. Priority Scheduling

  • Каждому процессу присваивается число-приоритет.
  • ЦП получает процесс с наивысшим приоритетом (обычно меньшее число = выше приоритет).
  • Бывает preemptive и non-preemptive.
  • SJF — частный случай Priority Scheduling, где приоритет = 1/(ожидаемая длина burst).
Проблема — starvation: процессы с низким приоритетом могут никогда не получить ЦП.
Решение — aging: со временем приоритет ожидающего процесса повышается.

7. Round Robin (RR)

  • Каждый процесс получает небольшой квант времени q (обычно 10–100 мс).
  • По истечении кванта процесс вытесняется и попадает в конец ready-очереди.
  • При n процессах и кванте q каждый получает 1/n ЦП кусками не больше q. Ни один не ждёт дольше (n−1)q.
  • Таймер генерирует прерывание каждые q для запуска планировщика.

Выбор q:

  • q большое → поведение приближается к FCFS;
  • q малое → растёт доля времени на переключение контекста;
  • правило: q ≫ context switch; обычно q ≈ 10–100 мс, переключение контекста < 10 мкс;
  • эмпирическое правило: 80% CPU-бёрстов должны быть короче q.

RR обычно даёт худший turnaround, чем SJF, но лучший response time.

8. Многоуровневые очереди

Multilevel Queue

  • Ready-очередь разбивается на несколько очередей, например:
    • foreground (интерактивные) — RR;
    • background (пакетные) — FCFS.
  • Процесс навсегда остаётся в своей очереди.
  • Нужно правило выбора между очередями:
    • Fixed priority: сначала foreground, потом background — рискует starvation;
    • Time slice: каждой очереди — свой процент ЦП (80% foreground / 20% background).

Multilevel Feedback Queue

  • Процесс может перемещаться между очередями — это естественный способ реализовать aging.
  • Параметры алгоритма: число очередей, алгоритм каждой, правила повышения, понижения и распределения новых процессов.

Пример с тремя очередями:

  • Q₀ — RR с квантом 8 мс;
  • Q₁ — RR с квантом 16 мс;
  • Q₂ — FCFS.

Новая задача попадает в Q₀. Если не успела за 8 мс — спускается в Q₁ (получает там ещё 16 мс). Если и там не завершилась — попадает в Q₂.

10. Многопроцессорное планирование

SMP, AMP, processor affinity

  • Asymmetric multiprocessing (AMP): только один процессор обращается к системным структурам данных — нет нужды в их разделении.
  • Symmetric multiprocessing (SMP): каждый процессор сам себя планирует; либо общая ready-очередь, либо своя у каждого. Самый распространённый подход сегодня.
  • Processor affinity: процесс «привязан» к процессору, на котором он сейчас работает — это сохраняет тёплый кэш.
    • Soft affinity — ОС старается, но может перенести;
    • Hard affinity — ОС обязана соблюдать.
  • Возможны processor sets — группы ЦП с собственной политикой.

NUMA

В NUMA-системах память «ближе» к одним ЦП и «дальше» от других. Хороший планировщик учитывает affinity не только для кэша, но и для расположения памяти процесса.

Балансировка нагрузки в SMP

  • Push migration: периодическая задача проверяет загрузку ЦП и при перекосе проталкивает задачу с перегруженного на менее загруженный.
  • Pull migration: простаивающий процессор сам забирает задачу из очереди занятого.

Многоядерные процессоры

  • Несколько ядер на одном физическом кристалле — быстрее и потребляют меньше энергии.
  • На каждом ядре растёт число аппаратных потоков.
  • Аппаратные потоки используют простои на ожидании памяти (memory stall): пока одно загружает данные, другое продолжает работать.

11. Real-Time CPU Scheduling

Soft vs Hard real-time

  • Soft real-time: нет гарантий — критичный процесс когда-нибудь будет запланирован, но без обещания «вовремя».
  • Hard real-time: задача обязана быть обслужена к дедлайну.

Виды задержек

  • Interrupt latency — от прихода прерывания до начала его обработчика.
  • Dispatch latency — пока планировщик снимает текущий процесс и запускает другой.
    • Conflict phase: вытеснение процесса, работающего в kernel mode; освобождение ресурсов, нужных более приоритетной задаче.

Периодические задачи

У периодической задачи: время обработки t, дедлайн d, период p, причём 0 ≤ t ≤ d ≤ p. Частота — 1/p.

Rate-Monotonic Scheduling (RMS)

Приоритет — обратно пропорционален периоду: чем меньше период, тем выше приоритет. Если P1 имеет меньший период, чем P2 — P1 приоритетнее.

Earliest Deadline First (EDF)

Приоритет назначается по дедлайну: чем раньше дедлайн, тем выше приоритет. Может уложиться в случаях, где RMS пропускает дедлайны.

Proportional Share

Система имеет T «долей»; приложение получает N < T долей и тем самым гарантированно получает N/T процессорного времени.

POSIX Real-Time

  • SCHED_FIFO — FCFS с FIFO-очередью, без time-slicing для потоков одного приоритета.
  • SCHED_RR — то же, но с разрезанием времени для потоков одного приоритета.
  • API: pthread_attr_getschedpolicy / setschedpolicy.

Виртуализация

Гипервизор планирует гостевые ОС на физические ЦП; каждая гостевая ОС, в свою очередь, планирует свои процессы, не зная, что не владеет ЦП полностью. Это может ухудшить response time, исказить часы в гостях и обесценить тонкую настройку их собственных планировщиков.

12. Примеры ОС

Linux: до 2.5 и O(1) (ядра 2.5–2.6)

  • До 2.5 — вариация классического UNIX-планировщика.
  • 2.5 — планировщик с O(1)-временем: preemptive, по приоритетам.
  • Два диапазона приоритетов: real-time (0–99) и nice (100–139). Меньшее число — выше приоритет.
  • Высокому приоритету — больший квант.
  • Две массива по приоритетам: active и expired. Когда active пустеет, массивы меняются местами.
  • Недостаток: плохой response time для интерактивных процессов.

Linux: CFS (с 2.6.23)

  • Completely Fair Scheduler.
  • Введены scheduling classes со своими приоритетами; берётся задача с наивысшим приоритетом из наивысшего класса. По умолчанию есть default и real-time.
  • Квант не фиксирован, а вычисляется как доля ЦП. Базируется на nice-значении от −20 до +19 (меньше = выше).
  • Target latency — интервал, за который каждая активная задача должна получить ЦП хотя бы один раз; растёт с числом активных задач.
  • Для каждой задачи поддерживается vruntime (виртуальное время), которое для задач низкого приоритета растёт быстрее.
  • Следующей выполняется задача с минимальным vruntime.
  • Real-time планируется по POSIX.1b со статическими приоритетами; nice −20 → глобальный приоритет 100, nice +19 → 139.

Windows

  • Preemptive, priority-based; высшая приоритетная задача выполняется первой.
  • Поток работает, пока (1) не заблокируется, (2) не израсходует квант, (3) не будет вытеснен более приоритетным.
  • 32 уровня приоритета: variable class 1–15, real-time class 16–31; priority 0 — поток управления памятью.
  • На каждый приоритет своя очередь; если ничего готово нет — выполняется idle-поток.
  • Классы приоритетов Win32: REALTIME, HIGH, ABOVE_NORMAL, NORMAL, BELOW_NORMAL, IDLE. Внутри класса — relative priority: TIME_CRITICAL, HIGHEST, ABOVE_NORMAL, NORMAL, BELOW_NORMAL, LOWEST, IDLE.
  • Если поток исчерпал квант — приоритет понижается, но не ниже базового. После ожидания — приоритет повышается (boost) в зависимости от того, чего ждали. Foreground-окно получает ×3 boost.
  • Windows 7: User-Mode Scheduling (UMS) — приложение само создаёт и планирует потоки в обход ядра. Эффективно при больших числах потоков. Пример: C++ ConcRT.

Solaris

  • Priority-based, шесть классов: Time Sharing (TS, по умолчанию), Interactive (IA), Real-Time (RT), System (SYS), Fair Share (FSS), Fixed Priority (FP).
  • Поток принадлежит ровно одному классу; у каждого класса свой алгоритм.
  • Time Sharing реализован как multilevel feedback queue; таблица диспетчера загружаема и настраивается админом.
  • Все классы конвертируются в общий глобальный приоритет; следующим выполняется поток с наибольшим. Несколько потоков одного приоритета обслуживаются по RR.

13. Оценка алгоритмов

Детерминированное моделирование

Берётся конкретная нагрузка, и для каждого алгоритма точно считается результат. Просто и быстро, но применимо только к этим входным данным.

Например, для 5 процессов, пришедших одновременно: FCFS даёт 28 мс среднего ожидания, non-preemptive SJF — 13 мс, RR — 23 мс.

Очереди и формула Литтла

Поведение процессов описывается вероятностно — обычно экспоненциальное распределение интервалов прихода и обслуживания. Систему моделируют как сеть серверов с очередями.

n = λ · W

где n — средняя длина очереди, λ — средняя интенсивность прибытия, W — среднее время ожидания. Закон Литтла верен для любого алгоритма и любого распределения прихода.

Пример. Если в среднем приходит 7 процессов в секунду и обычно в очереди 14 процессов — среднее ожидание = 2 секунды.

Симуляция

  • Очередевые модели ограничены — симуляция точнее.
  • Программная модель системы; время — переменная; собирается статистика.
  • Входные данные: генератор случайных чисел по заданному распределению или trace tapes — записи реальных событий из реальной системы.

Реализация

Даже симуляция ограничена в точности. В конце концов — собрать планировщик и протестировать в реальной системе. Высокая стоимость, высокий риск, окружения сильно отличаются. Поэтому гибкие планировщики позволяют настраивать поведение под систему или площадку, либо предоставляют API для управления приоритетами.

Главный вывод: ни один алгоритм планирования не оптимален во всех сценариях. FCFS прост, но даёт convoy effect; SJF оптимален по waiting time, но требует прогноза; RR хорош для интерактивности; многоуровневые feedback-очереди и CFS — компромиссы, объединяющие плюсы. Real-time требует отдельных гарантий (RMS, EDF). Оценивать алгоритм нужно под конкретную нагрузку — через моделирование, симуляцию или прямое внедрение.

1. Basic concepts and the CPU–I/O burst cycle

  • Maximum CPU utilization comes from multiprogramming.
  • Process execution is a cycle: CPU burst → I/O wait → CPU burst → I/O wait → …
  • A CPU burst is followed by an I/O burst; scheduling cares mainly about the distribution of CPU burst lengths.
Observation: the histogram of CPU burst times is typically strongly skewed toward short bursts — short bursts are far more common than long ones. This is precisely why SJF is effective in theory.

2. Short-term scheduler and dispatcher

The short-term scheduler picks the next ready process to receive the CPU. The ready queue may be ordered in many ways.

A scheduling decision happens when a process:

  1. switches from running to waiting (e.g., awaits I/O);
  2. switches from running to ready (e.g., on interrupt);
  3. switches from waiting to ready (I/O completed);
  4. terminates.

Scheduling under cases 1 and 4 is nonpreemptive. All other cases are preemptive. Preemption raises issues: shared-data access, preemption in kernel mode, interrupts during critical OS activities.

Dispatcher

The dispatcher hands the CPU to the process the scheduler chose. This involves:

  • context switching,
  • switching to user mode,
  • jumping to the right location in the user program to resume it.

Dispatch latency = time it takes to stop one process and start another.

3. Scheduling criteria

CPU utilization
Keep the CPU as busy as possible.
Throughput
Processes finished per unit time.
Turnaround time
Total time to execute a process.
Waiting time
Time spent in the ready queue.
Response time
Time from request to first response (not full output). Matters in interactive systems.

Optimization goals: maximize utilization and throughput; minimize turnaround, waiting, response time.

4. First-Come, First-Served (FCFS)

The simplest algorithm: whoever arrives first is served first.

Example 1. Arrival order: P1(24), P2(3), P3(3).

P10–24
P224–27
P327–30

Waiting times: P1=0, P2=24, P3=27. Average: (0+24+27)/3 = 17.

Example 2. Arrival order: P2, P3, P1.

P20–3
P33–6
P16–30

Average wait: (6+0+3)/3 = 3 — much better.

Convoy effect: short processes get stuck behind one long process. FCFS is especially bad with one CPU-bound and many I/O-bound processes.

5. Shortest-Job-First (SJF) and SRTF

  • Each process has a length for its next CPU burst.
  • Pick the process with the shortest predicted next burst.
  • SJF is optimal: it minimizes average waiting time for a given workload.
  • Problem: we don't know the next burst length — we have to estimate.

Estimating the next CPU burst

Use exponential averaging of previous bursts:

τn+1 = α · tn + (1 − α) · τn

where tn is the latest actual burst, τn the previous estimate, commonly α = ½.

  • α = 0: τn+1 = τn — recent history is ignored.
  • α = 1: τn+1 = tn — only the last burst counts.
  • Expanded: τn+1 = α·tn + (1−α)·α·tn−1 + … — each older sample weighs exponentially less.

Shortest-Remaining-Time-First (SRTF)

The preemptive version of SJF: if a new arrival has a shorter remaining burst than the running process, preempt. Average waiting drops further.

6. Priority Scheduling

  • Each process gets an integer priority.
  • The CPU goes to the highest-priority process (usually lower number = higher priority).
  • Can be preemptive or nonpreemptive.
  • SJF is a special case where priority = inverse of predicted next CPU burst.
Problem — starvation: low-priority processes may never run.
Solution — aging: raise the priority of a process the longer it has waited.

7. Round Robin (RR)

  • Each process gets a small time quantum q (typically 10–100 ms).
  • When the quantum expires, the process is preempted and goes to the back of the ready queue.
  • With n processes and quantum q, each gets 1/n of the CPU in chunks of at most q. No process waits more than (n−1)q.
  • A timer interrupt every q kicks the scheduler.

Choosing q:

  • q large → behaves like FCFS;
  • q small → context-switch overhead dominates;
  • rule: q ≫ context switch; commonly q ≈ 10–100 ms, context switch < 10 µs;
  • rule of thumb: 80% of CPU bursts should be shorter than q.

RR typically gives worse turnaround than SJF but better response.

8. Multilevel queues

Multilevel Queue

  • The ready queue is split into several queues, e.g.:
    • foreground (interactive) — RR;
    • background (batch) — FCFS.
  • A process stays permanently in its assigned queue.
  • Need a policy across queues:
    • Fixed priority: serve foreground first, then background — risks starvation.
    • Time slice: each queue gets a share of CPU (e.g. 80% / 20%).

Multilevel Feedback Queue

  • Processes can move between queues — a natural way to do aging.
  • Parameters: number of queues, algorithm per queue, rules for promotion, demotion, and initial placement.

Three-queue example:

  • Q₀ — RR with 8 ms quantum;
  • Q₁ — RR with 16 ms quantum;
  • Q₂ — FCFS.

A new job enters Q₀. If it doesn't finish in 8 ms, it drops to Q₁ (gets 16 more ms). If it still doesn't finish, it drops to Q₂.

10. Multiprocessor scheduling

SMP, AMP, processor affinity

  • Asymmetric multiprocessing (AMP): only one processor touches system data structures — avoids data sharing.
  • Symmetric multiprocessing (SMP): every processor schedules itself; either a single shared ready queue or one per CPU. The common case today.
  • Processor affinity: a process is biased toward the CPU it last ran on — preserves cache warmth.
    • Soft affinity — best effort;
    • Hard affinity — the OS must obey.
  • Also processor sets — groups of CPUs with their own policy.

NUMA

In NUMA systems memory is closer to some CPUs and farther from others. A good scheduler considers affinity not only for cache but for memory placement.

Load balancing in SMP

  • Push migration: a periodic task checks per-CPU load and pushes tasks off overloaded CPUs.
  • Pull migration: an idle CPU pulls a waiting task from a busy one.

Multicore processors

  • Multiple cores on one chip — faster, lower power.
  • Hardware threads per core also growing.
  • Hardware threads exploit memory stalls: while one waits on memory, another makes progress.

11. Real-Time CPU Scheduling

Soft vs hard real-time

  • Soft real-time: no guarantee — the critical process will run eventually, but no "on time" promise.
  • Hard real-time: the task must be serviced by its deadline.

Latency types

  • Interrupt latency — arrival of interrupt → start of its handler.
  • Dispatch latency — scheduler stopping the current task → starting another.
    • Conflict phase: preempting a process running in kernel mode; waiting for a low-priority process to release resources needed by a high-priority one.

Periodic tasks

A periodic task has processing time t, deadline d, period p, with 0 ≤ t ≤ d ≤ p. Its rate is 1/p.

Rate-Monotonic Scheduling (RMS)

Priority is inversely proportional to the period: shorter period → higher priority. If P1 has a shorter period than P2, P1 has higher priority.

Earliest Deadline First (EDF)

Priority is set by deadline: the earlier the deadline, the higher the priority. EDF can meet deadlines that RMS misses.

Proportional Share

The system has T shares; an application gets N < T shares and is therefore guaranteed N/T of CPU time.

POSIX Real-Time

  • SCHED_FIFO — FCFS via a FIFO queue, no time-slicing for equal-priority threads.
  • SCHED_RR — like SCHED_FIFO but with time-slicing for equal-priority threads.
  • API: pthread_attr_getschedpolicy / setschedpolicy.

Virtualization

The hypervisor schedules guest OSes onto physical CPUs; each guest in turn schedules its own processes, unaware it doesn't really own the CPUs. This can degrade response time, distort guest clocks, and undo the guest's own scheduling decisions.

12. OS examples

Linux: pre-2.5 and the O(1) scheduler (2.5–2.6)

  • Before 2.5 — a variation of the classic UNIX scheduler.
  • 2.5 introduced the O(1) scheduler: preemptive, priority-based.
  • Two priority ranges: real-time (0–99) and nice (100–139). Lower number = higher priority.
  • Higher priority gets a larger quantum.
  • Two arrays per CPU runqueue: active and expired. When active drains, the two are swapped.
  • Drawback: poor response time for interactive processes.

Linux: CFS (since 2.6.23)

  • Completely Fair Scheduler.
  • Scheduling classes with their own priorities; the scheduler picks the highest-priority task in the highest class. By default: default and real-time.
  • Instead of fixed quanta, allocates a proportion of CPU time. Based on nice values from −20 to +19 (lower = higher).
  • Target latency — the interval within which each active task should run at least once; grows with the number of active tasks.
  • Each task has a vruntime (virtual run time); lower-priority tasks accumulate vruntime faster.
  • The next task is the one with the smallest vruntime.
  • Real-time scheduling follows POSIX.1b with static priorities; nice −20 maps to global priority 100, nice +19 to 139.

Windows

  • Preemptive, priority-based; highest-priority thread runs next.
  • A thread runs until (1) it blocks, (2) its quantum expires, or (3) a higher-priority thread preempts it.
  • 32 priority levels: variable class 1–15, real-time class 16–31; priority 0 is the memory-management thread.
  • One queue per priority; if no thread is runnable, the idle thread runs.
  • Win32 priority classes: REALTIME, HIGH, ABOVE_NORMAL, NORMAL, BELOW_NORMAL, IDLE. Within a class, a relative priority: TIME_CRITICAL, HIGHEST, ABOVE_NORMAL, NORMAL, BELOW_NORMAL, LOWEST, IDLE.
  • When a quantum expires the priority is lowered, but never below the base. After a wait the priority is boosted based on what was waited on. The foreground window gets a 3× boost.
  • Windows 7: User-Mode Scheduling (UMS) — applications create and manage threads independently of the kernel. Much more efficient for huge thread counts. Example: C++ ConcRT.

Solaris

  • Priority-based with six classes: Time Sharing (TS, default), Interactive (IA), Real-Time (RT), System (SYS), Fair Share (FSS), Fixed Priority (FP).
  • A thread is in exactly one class at a time; each class has its own algorithm.
  • Time Sharing uses multilevel feedback queues; the dispatch table is loadable and admin-tunable.
  • All classes map into a single global priority; the highest runs next. Threads at the same priority are scheduled RR.

13. Algorithm evaluation

Deterministic modeling

Take a particular workload and compute exact results for each algorithm. Simple and fast — but only valid for those inputs.

For example, with 5 processes arriving at time 0: FCFS gives 28 ms average wait, non-preemptive SJF gives 13 ms, RR gives 23 ms.

Queueing models and Little's Law

Describe processes probabilistically — typically exponential arrival and service intervals. The system is modeled as a network of servers with queues.

n = λ · W

where n = average queue length, λ = average arrival rate, W = average wait. Little's Law holds for any scheduling algorithm and any arrival distribution.

Example. If 7 processes arrive per second on average and there are normally 14 in the queue, average wait = 2 seconds.

Simulation

  • Queueing models are limited — simulation is more accurate.
  • A programmed model of the computer system; time is a variable; statistics are gathered.
  • Input data come from a probabilistic random-number generator, mathematically defined distributions, or trace tapes — records of real events from real systems.

Implementation

Even simulation has limited accuracy. Ultimately — build the scheduler and run it on a real system. High cost, high risk, and environments vary. So the most flexible schedulers can be tuned per-site or per-system, or expose APIs for priority adjustment.

Take-away: no single scheduling algorithm wins in every scenario. FCFS is simple but suffers the convoy effect; SJF is optimal for waiting time but needs prediction; RR is great for interactivity; multilevel-feedback queues and CFS are compromises that combine the strengths. Real-time needs separate guarantees (RMS, EDF). The right algorithm depends on the workload — measured via modeling, simulation, or live deployment.
