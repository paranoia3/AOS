Network and Distributed Systems

Сети и распределённые системы — резюме и пояснения

Эта лекция объясняет, что такое распределённая система, зачем она нужна (sharing, speedup, reliability), как устроены LAN/WAN, что такое 7 уровней OSI и стек TCP/IP, как пакет идёт от отправителя к получателю (DNS → IP → ARP → MAC), чем отличаются UDP и TCP, что такое сетевая и распределённая ОС, как обеспечиваются robustness, transparency, scalability, и как устроены распределённые файловые системы (client-server vs cluster-based, GFS/HDFS, политика кэширования и согласованности).

1. Что такое распределённая система

Распределённая система — набор слабо связанных узлов (nodes), соединённых коммуникационной сетью. У каждого узла своя память и свой такт; общение — только через передачу сообщений.
  • Узлы называют процессорами, компьютерами, машинами, хостами.
  • Site — физическое местоположение машины; node — конкретная система.
  • Обычно server предоставляет ресурс, который хочет использовать client.
  • Конфигурации: client-server, peer-to-peer, гибрид.
    • В P2P каждый узел и клиент, и сервер.

2. Зачем нужны распределённые системы

Resource sharing
Файлы, печать, удалённые БД, GPU и другое спецоборудование.
Computation speedup
Распределить подзадачи по узлам; load balancing — перенос задач на менее загруженные узлы.
Reliability
Детектировать и восстанавливаться после отказа узла, передать функции, ре-интегрировать.

3. LAN и WAN

Local-Area Network (LAN)

  • Небольшая географическая область.
  • Компьютеры, ноутбуки, мобильные устройства + периферия (принтеры, СХД) + роутеры.
  • Ethernet (IEEE 802.3): от 10 Mbps до 10+ Gbps.
  • Wi-Fi (IEEE 802.11): от 11 Mbps до 400+ Mbps.

Wide-Area Network (WAN)

  • Связывает географически удалённые сайты point-to-point-каналами.
  • Среды: телефон, выделенные линии, оптика, микроволновка, радио, спутник.
  • Реализуется через роутеры.
  • Магистральные провайдеры — 40–100 Gbps; локальные ISP медленнее.
  • Интернет — самый известный WAN.

4. Именование и DNS

  • У каждой системы — уникальное имя; у каждого процесса — уникальный process-id.
  • Процессы на удалённых машинах идентифицируются парой <host-name, identifier>.
  • DNS (Domain Name System) задаёт иерархическую структуру имён и преобразует имена в адреса.

OSI — формализация ранних работ конца 1970-х. Не получила широкого распространения. На практике царит TCP/IP — у него меньше уровней, он сочетает несколько функций в одном слое; в теории сложнее имплементируется, но эффективнее.

6. TCP/IP, ARP, MAC

Маршрутизация

  • У каждого хоста есть имя и связанный IP-адрес (host-id), иерархический и сегментированный.
  • Отправитель смотрит свои маршрутные таблицы и находит роутер.
  • Роутер по сегменту host-id решает, куда передать пакет — возможно, через несколько роутеров.
  • Пакет либо самостоятелен, либо пересобирается из нескольких на приёмнике.

Внутри сети: MAC и ARP

  • У каждого Ethernet/Wi-Fi устройства есть MAC-адрес (Medium Access Control).
  • Два устройства в одном LAN общаются по MAC.
  • Чтобы узнать соответствие IP → MAC, используется ARP (Address Resolution Protocol).
  • Broadcast — специальный сетевой адрес, на который реагируют все хосты сети; не пересылается роутерами в другие сети.

7. UDP и TCP

Когда пакет дошёл до хоста по IP, его нужно отдать правильному процессу. Это делают транспортные протоколы через номера портов.

Хорошо известные порты:

  • FTP — 21
  • ssh — 22
  • SMTP — 25
  • HTTP — 80

UDP (User Datagram Protocol)

  • «Голое» расширение IP портами.
  • Ненадёжен: пакеты могут потеряться или прийти не по порядку.
  • Connectionless: нет рукопожатия и закрытия.
  • UDP-пакеты называют datagrams.

TCP (Transmission Control Protocol)

  • Надёжен и connection-oriented.
  • Поверх ненадёжной сети даёт абстракцию упорядоченного, непрерывного байтового потока.
  • Каждый пакет получает ACK; при отсутствии ACK по таймеру — повторная отправка.
  • Sequence numbers позволяют упорядочить и выявить пропавшие пакеты.
  • Соединение открывается three-way handshake, закрывается серией контрольных пакетов.
  • Получатель может отправлять cumulative ACK за серию пакетов; отправитель может слать несколько пакетов до ACK ради throughput.
  • Flow control — не перегружать получателя.
  • Congestion control — оценивать загруженность сети и регулировать скорость.

8. Сетевая vs распределённая ОС

Network Operating Systems

  • Пользователь осознаёт множественность машин.
  • Доступ к ресурсам — явный:
    • remote login (ssh kristen.cs.yale.edu),
    • передача файлов через FTP,
    • облачное хранилище — загрузка, скачивание, шаринг.
  • Пользователь должен сменить парадигму: открыть сессию, вводить сетевые команды.
  • Сложнее в использовании.

Distributed Operating Systems

Пользователь не подозревает о множественности машин — удалённые ресурсы выглядят как локальные.

  • Data migration — передаётся целый файл или только нужная часть.
  • Computation migration — переносится вычисление, а не данные (через RPC или сообщения).
  • Process migration — выполнение процесса или его частей на разных сайтах:
    • load balancing,
    • computation speedup — параллелизм,
    • предпочтения по железу/софту,
    • data access — лучше запустить процесс там, где данные.
  • WWW — пример такого подхода.

9. Robustness и обнаружение отказов

Типы отказов: link failure, site failure, потеря сообщения. Fault-tolerant система выдерживает определённый уровень отказов. Процесс: detection → reconfiguration → recovery.

Обнаружение отказа — heartbeat protocol

  1. Сайты A и B установили связь.
  2. С фиксированным интервалом обмениваются «I-am-up» сообщениями.
  3. Если A не получает сообщение от B в срок — возможно, B упал или сообщение потеряно.
  4. A посылает «Are-you-up?».
  5. Без ответа — повторить или попробовать альтернативный маршрут.
  6. Если ответа нет — значит, произошёл какой-то отказ.
Главная сложность: A не может точно определить, что именно случилось — упал B, упал прямой канал, упал альтернативный канал, или потеряно сообщение.

Reconfiguration и Recovery

  • Если упал канал A↔B — сообщить всем сайтам.
  • Если упал сайт — все остальные узнают, что его сервисы недоступны.
  • Когда узел/канал возвращается — снова рассылается broadcast.

10. Transparency и Scalability

Transparency

  • Распределённая система должна выглядеть как обычная централизованная.
  • UI не отличает локальные и удалённые ресурсы. Пример: NFS.
  • User mobility: пользователь логинится с любой машины и видит своё окружение. Пример: LDAP + виртуализация рабочих столов.

Scalability

  • При росте нагрузки система должна легко принимать новые ресурсы.
  • Реагирует на рост нагрузки плавно.
  • Добавляя ресурсы, надо учитывать косвенную нагрузку на другие компоненты.
  • Сжатие и дедупликация экономят место и сетевые ресурсы.

11. Распределённая файловая система (DFS)

DFS — файловая система, у которой клиенты, серверы и устройства хранения разбросаны по машинам распределённой системы. Для клиента она должна выглядеть как обычная централизованная ФС. Главное отличие — управление рассредоточенными хранилищами.
  • Service — программа на одной или нескольких машинах, предоставляющая функциональность априори неизвестным клиентам.
  • Server — реализация service на одной машине.
  • Client — процесс, вызывающий service через client interface (для файла — create, delete, read, write).
  • Клиентский интерфейс DFS должен быть прозрачным — не отличать локальные и удалённые файлы.

Ключевые вызовы: именование и прозрачность, удалённый доступ, кэширование и согласованность.

12. Client-Server vs Cluster-based DFS

Client-Server DFS

  • Серверы хранят файлы и метаданные на подключённых хранилищах.
  • Клиенты запрашивают файлы у сервера.
  • Сервер отвечает за аутентификацию, проверку прав, доставку.
  • Изменения клиента отправляются обратно на сервер.
  • Примеры: NFS, OpenAFS.
  • Проблемы: single point of failure, узкое место для всех запросов, ограничения по масштабу и пропускной способности.

Cluster-Based DFS

  • Создан более отказоустойчивым и масштабируемым.
  • Примеры: GFS (Google File System), HDFS (Hadoop).
  • Клиенты подключаются к master metadata server и нескольким data servers, хранящим chunks (порции файлов).
  • Master хранит маппинг chunk→data server и иерархию каталогов/файлов.
  • Каждый chunk реплицирован n раз.

Принципы дизайна GFS:

  • Отказы железа — норма, а не исключение.
  • Файлы очень большие.
  • В основном append в конец, а не перезапись.
  • Можно изменить API файловой системы → больше гибкости (приложения переписываются под новый API).
  • MapReduce — модульный слой поверх GFS для крупномасштабных параллельных вычислений; Hadoop тоже стекируется и модульный.

13. Именование в DFS

  • Naming — соответствие логических и физических объектов.
  • Multilevel mapping — абстракция файла, скрывающая, где и как он хранится на диске.
  • Прозрачная DFS скрывает, где в сети лежит файл.
  • Для реплицированных файлов отображение возвращает множество локаций; и сами копии, и их места скрыты.

Структуры именования

СвойствоОписание
Location transparencyИмя файла не раскрывает физическое расположение.
Location independenceИмя файла не нужно менять при переносе файла.

Большинство DFS используют статическое, location-transparent отображение для имён уровня пользователя. Некоторые поддерживают миграцию файлов (OpenAFS). Hadoop поддерживает миграцию, но без соблюдения POSIX. Amazon S3 предоставляет блоки хранилища через API и динамически размещает данные.

Три подхода к схемам именования

  1. Имя = <host-name, local-name> — глобально уникальное, но ни transparent, ни independent.
  2. Монтирование удалённых каталогов в локальные — выглядит как единое дерево, но прозрачно только для уже смонтированных (стиль NFS).
  3. Единое глобальное имя для всей системы. Если сервер недоступен — какой-то произвольный набор каталогов на разных машинах становится недоступным.

14. Удалённый доступ и кэширование

После того как сервер найден через схему имён, нужно передать данные.

Remote-service механизм

  • Запросы доставляются на сервер; сервер выполняет и возвращает результат.
  • Самая распространённая реализация — RPC.

Кэширование

  • Снижает сетевой трафик: недавно прочитанные блоки оседают в кэше.
  • Если данных нет в кэше — копия тащится с сервера и кладётся в кэш; доступ идёт к копии.
  • Master-копия — на сервере, копии (частей файла) — в разных кэшах.
  • Возникает cache-consistency problem — поддержание согласованности.
  • Иногда это называют network virtual memory.

Где располагать кэш — диск или RAM

Disk cacheMain-memory cache
Надёжнее, переживает рестартПозволяет diskless-рабочие станции
Не надо фетчить заново после восстановленияБыстрый доступ, прирост в больших памятях

Кэши сервера (для ускорения дискового I/O) всегда в памяти; кэш в памяти у клиента — единая модель для сервера и клиента.

Политики обновления кэша

  • Write-through — сразу пишет данные на сервер. Надёжно, но медленно.
  • Delayed-write (write-back) — пишет в кэш, потом «когда-нибудь» на сервер. Быстро, но при крахе клиента данные теряются. Бывают варианты:
    • периодический скан кэша и flush изменённых блоков;
    • write-on-close — отправка данных на сервер при закрытии файла. Хорошо подходит для файлов, открытых долго и часто меняемых.

15. Согласованность кэша

Главный вопрос: совпадает ли локальная копия с master-копией?

ПодходКто проверяет
Client-initiatedКлиент запускает проверку; сервер сравнивает локальные данные с мастером.
Server-initiatedСервер запоминает, какие части файлов кэшированы у каких клиентов; при потенциальной несогласованности сам реагирует.

В cluster-based DFS

  • Сложнее из-за метаданных и реплик.
  • HDFS — только append-only, один писатель → проще согласованность.
  • GFS — допускает random writes с параллельными писателями → сложнее гарантии записи.
Главный вывод: распределённая система меняет «локальную ОС» на коллекцию слабосвязанных узлов, общающихся по сети. Чтобы это работало, нужны слои: физический ↔ data-link ↔ network (IP+ARP+MAC) ↔ transport (UDP/TCP) ↔ выше. Дальше всё упирается в три фундаментальных вопроса: robustness (как переживать отказы), transparency (как скрыть «распределённость» от пользователя) и scalability (как расти). DFS — конкретное приложение этих принципов: client-server прост и хорошо знаком, cluster-based (GFS/HDFS) рассчитан на отказы железа и огромные файлы.

1. What a distributed system is

A distributed system is a collection of loosely coupled nodes interconnected by a communications network. Each node has its own memory and clock; communication happens only by message passing.
  • Nodes are variously called processors, computers, machines, hosts.
  • Site is the physical location; node is a specific system.
  • Usually a server has a resource a client wants.
  • Configurations: client-server, peer-to-peer, hybrid.
    • In P2P, every node is both client and server.

2. Why distribute

Resource sharing
Files, printing, remote databases, GPUs and other special hardware.
Computation speedup
Split work across nodes; load balancing moves jobs to lighter-loaded sites.
Reliability
Detect and recover from a site failure, hand off functions, re-integrate the failed site.

3. LAN and WAN

Local-Area Network (LAN)

  • Small geographical area.
  • Computers, laptops, mobile devices + peripherals (printers, storage) + routers.
  • Ethernet (IEEE 802.3): 10 Mbps → 10+ Gbps.
  • Wi-Fi (IEEE 802.11): 11 Mbps → 400+ Mbps.

Wide-Area Network (WAN)

  • Connects geographically separated sites via point-to-point links.
  • Media: telephone lines, leased lines, optical, microwave, radio, satellite.
  • Routed via routers.
  • Backbones at 40–100 Gbps; local ISPs slower.
  • The Internet is the most famous WAN.

4. Naming and DNS

  • Every system has a unique name; every process has a unique process-id.
  • Processes on remote machines are identified by <host-name, identifier>.
  • DNS (Domain Name System) defines the hierarchical naming structure and resolves names to addresses.

5. The seven OSI layers

7. Application
user-facing: FTP, remote login, email, distributed DB schemas
6. Presentation
format conversion, half/full duplex
5. Session
process-to-process session protocols
4. Transport
packetizing, ordering, flow control, physical addresses
3. Network
routing, addressing, route tables
2. Data-link
frames, error detection/recovery
1. Physical
mechanical and electrical transmission of bits

OSI formalizes earlier 1970s work but isn't widely deployed. The dominant stack is TCP/IP — fewer layers, combining several functions per layer; harder to implement in theory but more efficient.

6. TCP/IP, ARP, MAC

Routing

  • Every host has a name and an associated IP address (host-id), hierarchical and segmented.
  • The sender consults its routing tables to find a router.
  • The router uses the segmented host-id to decide where to forward — possibly through several routers.
  • A packet may be a whole message, or must be reassembled from many at the destination.

Inside the LAN: MAC and ARP

  • Every Ethernet/Wi-Fi device has a MAC address (Medium Access Control).
  • Two devices on the same LAN talk by MAC.
  • ARP (Address Resolution Protocol) maps IP → MAC.
  • A broadcast uses a special address — every host receives it; routers don't forward broadcasts to other networks.

7. UDP and TCP

Once an IP packet reaches the host, it must be delivered to the right process. Transport protocols do this via port numbers.

Well-known ports:

  • FTP — 21
  • ssh — 22
  • SMTP — 25
  • HTTP — 80

UDP (User Datagram Protocol)

  • A bare-bones extension of IP with port numbers.
  • Unreliable: packets may be lost or reordered.
  • Connectionless: no setup, no tear-down.
  • UDP packets are called datagrams.

TCP (Transmission Control Protocol)

  • Reliable and connection-oriented.
  • Provides an in-order, uninterrupted byte stream across an unreliable network.
  • Every packet is ACK'd; on timeout the sender retransmits.
  • Sequence numbers reorder packets and detect losses.
  • Connections open with a three-way handshake and close with control packets.
  • Receivers can send cumulative ACKs; senders can pipeline packets before ACKs.
  • Flow control — don't overwhelm the receiver.
  • Congestion control — sense network load and adapt sending rate.

8. Network vs. distributed OS

Network Operating Systems

  • Users are aware of multiple machines.
  • Remote access is explicit:
    • remote login (ssh kristen.cs.yale.edu),
    • file transfer via FTP,
    • cloud storage — upload, download, share.
  • Users must shift paradigms — open a session, use network commands.
  • Harder to use.

Distributed Operating Systems

Users aren't aware of multiple machines — remote resources look local.

  • Data migration — transfer the whole file or just the parts needed.
  • Computation migration — move the computation, not the data (via RPC or messaging).
  • Process migration — run a process or parts of it at different sites:
    • load balancing,
    • speedup through concurrency,
    • hardware/software preferences,
    • data access — run the process where the data is.
  • The WWW is an example.

9. Robustness and failure detection

Failure types: link, site, message. A fault-tolerant system tolerates some level of failure. The flow: detection → reconfiguration → recovery.

Detection — heartbeat protocol

  1. Site A and site B establish a link.
  2. They exchange "I-am-up" messages at fixed intervals.
  3. If A misses one — B might be down or the message lost.
  4. A sends "Are-you-up?".
  5. If no reply, retry or try an alternate route.
  6. If still no reply — a failure has occurred.
The hard part: A cannot tell which failure exactly happened — B is down, the direct link is down, the alternate link is down, or the message was lost.

Reconfiguration and recovery

  • If link A↔B is down — broadcast to every site.
  • If a site is down — every other site learns its services are unavailable.
  • When the node/link returns — broadcast that too.

10. Transparency and scalability

Transparency

  • A distributed system should look like a centralized one.
  • The UI doesn't distinguish local from remote. Example: NFS.
  • User mobility: a user can log into any machine and see their environment. Example: LDAP plus desktop virtualization.

Scalability

  • As demand grows, the system must accept new resources easily.
  • It reacts gracefully to load.
  • Adding resources can create indirect load on other components — beware.
  • Compression and deduplication save storage and network.

11. Distributed File System (DFS)

A DFS is a file system whose clients, servers, and storage devices are dispersed across the machines of a distributed system. To the client it should look like an ordinary centralized FS. The distinguishing feature is managing dispersed storage.
  • Service — software on one or more machines providing functionality to clients unknown a priori.
  • Server — the service running on a single machine.
  • Client — a process that invokes a service via the client interface (for files: create, delete, read, write).
  • The DFS client interface should be transparent — local and remote files look the same.

Key challenges: naming and transparency, remote access, caching and consistency.

12. Client-server vs. cluster-based DFS

Client-Server DFS

  • Servers hold files and metadata on attached storage.
  • Clients request files from the server.
  • The server handles authentication, permission checks, delivery.
  • Client changes propagate back to the server.
  • Examples: NFS, OpenAFS.
  • Problems: single point of failure, bottleneck for all metadata and data requests, scalability and bandwidth limits.

Cluster-Based DFS

  • Designed to be more fault-tolerant and scalable.
  • Examples: GFS (Google File System), HDFS (Hadoop).
  • Clients connect to a master metadata server and several data servers holding file chunks.
  • The master maps chunks to data servers plus the directory/file hierarchy.
  • Each chunk is replicated n times.

GFS design principles:

  • Hardware failures are the norm, not exceptional.
  • Files are very large.
  • Most writes are appends, not overwrites.
  • The FS API can be redesigned for flexibility (apps must adopt it).
  • MapReduce is a modular layer on top of GFS for large-scale parallel computation; Hadoop is similarly stackable.

13. Naming in a DFS

  • Naming — mapping between logical and physical objects.
  • Multilevel mapping — a file abstraction that hides where and how the file is stored.
  • A transparent DFS hides the network location of the file.
  • For replicated files the mapping returns a set of locations; multiple copies and their locations are hidden from the user.

Naming properties

PropertyMeaning
Location transparencyThe name doesn't reveal the file's physical location.
Location independenceThe name doesn't have to change when the file moves.

Most DFSes use static, location-transparent mapping for user-level names. Some support file migration (OpenAFS). Hadoop supports it but without POSIX. Amazon S3 provides on-demand storage blocks via APIs, dynamically placing data.

Three naming schemes

  1. Name = <host-name, local-name> — globally unique, but neither transparent nor independent.
  2. Attach remote directories to local ones — appears as one tree, transparent only after mounting (NFS style).
  3. A single global name structure for the whole system. If a server is unavailable, an arbitrary set of directories on different machines becomes unavailable.

14. Remote access and caching

Once the naming scheme has found the server, data must be transferred.

Remote-service mechanism

  • Requests are delivered to the server; the server executes and returns the result.
  • Most common implementation: RPC.

Caching

  • Reduces network traffic — recently-used blocks live in a cache.
  • If data is missing, a copy is fetched from the server and cached; accesses use the cached copy.
  • The master copy is on the server; partial copies are scattered across caches.
  • This creates the cache-consistency problem.
  • Sometimes called network virtual memory.

Cache location — disk vs. memory

Disk cacheMain-memory cache
More reliable, survives rebootPermits diskless workstations
No need to refetch after recoveryFaster access; scales with memory size

Server caches (for disk I/O) always live in memory; using memory caches on clients gives one unified mechanism.

Cache update policies

  • Write-through — writes immediately reach the server. Reliable but slow.
  • Delayed-write (write-back) — writes hit the cache and propagate later. Fast, but data is lost if the client crashes. Variants:
    • periodic scan, flushing dirty blocks;
    • write-on-close — send back when the file is closed. Good for long-open, frequently-modified files.

15. Cache consistency

The key question: is the cached copy consistent with the master?

ApproachWho checks
Client-initiatedThe client triggers a validity check; the server compares with the master.
Server-initiatedThe server records what each client caches; when a potential inconsistency arises, it acts.

In cluster-based DFS

  • More complex due to metadata server and replicated chunks.
  • HDFS — append-only, single writer → consistency is simpler.
  • GFS — allows random writes with concurrent writers → write-consistency guarantees are harder.
Take-away: a distributed system replaces "one local OS" with a collection of loosely coupled nodes talking over a network. Making that work requires layered protocols: physical ↔ data-link ↔ network (IP+ARP+MAC) ↔ transport (UDP/TCP) ↔ higher. Then everything boils down to three fundamental questions: robustness (surviving failures), transparency (hiding distribution from users), and scalability (growing gracefully). DFS is a concrete instance: client-server is simple and familiar, cluster-based (GFS/HDFS) is designed for hardware failure as the norm and very large files.
