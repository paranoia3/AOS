Threads & Concurrency

Потоки и параллелизм — резюме и пояснения

Эта лекция объясняет, что такое поток, чем он отличается от процесса, какие преимущества даёт многопоточность, как соотносятся параллелизм и конкурентность, какие существуют модели отображения пользовательских потоков на ядерные, какие библиотеки потоков применяются на практике (Pthreads, Windows, Java), что такое неявная многопоточность (Thread Pools, OpenMP, GCD), и какие тонкие проблемы возникают при разработке многопоточных программ — fork/exec, сигналы, отмена потоков, TLS.

1. Что такое поток

Поток (thread) — это базовая единица использования ЦП. Каждый поток состоит из:

  • идентификатора потока (thread ID),
  • счётчика команд (PC),
  • набора регистров,
  • собственного стека.
Потоки одного процесса разделяют между собой: секцию кода, секцию данных, открытые файлы и сигналы — то есть ресурсы, принадлежащие процессу.

Традиционный процесс имеет один поток управления. Если процесс имеет несколько потоков управления, он может выполнять более одной задачи одновременно.

Пример. Текстовый редактор: один поток отображает графику, другой принимает нажатия клавиш, третий в фоне проверяет орфографию и грамматику.

2. Мотивация: зачем нужны потоки

  • Большинство современных приложений — многопоточные.
  • Несколько задач внутри приложения могут быть реализованы отдельными потоками: обновление UI, загрузка данных, проверка орфографии, обработка сетевых запросов.
  • Создание процесса — «тяжёлая» операция, создание потока — «лёгкая».
  • Многопоточность упрощает код и повышает эффективность.
  • Ядра современных ОС, как правило, сами многопоточные.

3. Преимущества многопоточности

Отзывчивость
Часть процесса может быть заблокирована, но программа продолжает работать. Особенно важно для пользовательских интерфейсов.
Разделение ресурсов
Потоки автоматически разделяют ресурсы процесса — это проще, чем разделяемая память или передача сообщений.
Экономия
Создание потока дешевле создания процесса; переключение между потоками — дешевле переключения контекста процессов.
Масштабируемость
Процесс может задействовать многопроцессорную/многоядерную архитектуру.

4. Многоядерное программирование

Многоядерные и многопроцессорные системы создают давление на программистов. Основные вызовы:

  • Разделение задач — на какие части разбить работу.
  • Балансировка — равная загрузка ядер.
  • Разделение данных — какие данные кому достаются.
  • Зависимости данных — синхронизация между потоками.
  • Тестирование и отладка — особенно сложны для многопоточных программ.

Типы параллелизма

ТипЧто делитЧто делают потоки
Data parallelismРаспределяет подмножества одних и тех же данных по ядрамКаждое ядро выполняет одну и ту же операцию
Task parallelismРаспределяет потоки/задачи по ядрамКаждый поток выполняет свою уникальную операцию

С ростом числа потоков растёт и архитектурная поддержка: процессоры имеют ядра и аппаратные потоки на каждом ядре. Например, Oracle SPARC T4: 8 ядер × 8 аппаратных потоков на ядро.

5. Параллелизм vs конкурентность

Конкурентность (concurrency) — несколько потоков делают прогресс, но не обязательно одновременно. Возможна даже на одноядерной системе — за счёт планировщика, который быстро переключается между потоками.

Параллелизм (parallelism) — несколько потоков действительно выполняются одновременно. Требует многоядерной системы с несколькими ЦП.

На одноядерной системе возможна только конкурентность. Параллелизм требует нескольких ЦП.

6. Закон Амдала

Определяет прирост производительности при добавлении дополнительных ядер к приложению, имеющему последовательную и параллельную части.

speedup = 1 / ( S + (1 − S) / N )

где S — последовательная часть программы, N — число ядер.

  • Если 75% программы параллельно, 25% последовательно, переход с 1 на 2 ядра даёт ускорение в 1.6 раза.
  • При N → ∞ ускорение стремится к 1 / S.
  • Вывод: последовательная часть оказывает непропорционально большое влияние на достижимое ускорение.

Замечание из лекции: учитывает ли этот закон современные многоядерные системы? — Это открытый вопрос: при большом числе ядер на первый план выходят кэш, межъядерные коммуникации, NUMA, и закон Амдала становится лишь верхней оценкой.

8. Модели многопоточности

Пользовательские потоки в итоге должны быть отображены на ядерные потоки, чтобы выполняться на ЦП. Существуют четыре основные модели.

Many-to-One (многие к одному)

User T₁User T₂User T₃→Kernel Thread
  • Много пользовательских потоков отображаются на один ядерный поток.
  • Если один поток блокируется — блокируются все.
  • Не может реально работать параллельно на многоядерной системе.
  • Примеры: Solaris Green Threads, GNU Portable Threads.

One-to-One (один к одному)

User T₁→Kernel T₁User T₂→Kernel T₂
  • Каждый пользовательский поток отображается на отдельный ядерный поток.
  • Больше конкурентности, чем M:1.
  • Накладные расходы могут ограничивать число потоков на процесс.
  • Примеры: Windows, Linux, Solaris ≥ 9.

Many-to-Many (многие ко многим)

User T₁User T₂User T₃→Kernel T₁Kernel T₂
  • Много пользовательских потоков отображается на несколько ядерных потоков.
  • ОС создаёт ровно столько ядерных потоков, сколько нужно.
  • Примеры: Solaris до версии 9, Windows + ThreadFiber package.

Two-level (двухуровневая)

Похожа на M:M, но дополнительно позволяет привязать пользовательский поток к конкретному ядерному. Примеры: IRIX, HP-UX, Tru64 UNIX, Solaris ≤ 8.

9. Библиотеки потоков

Библиотека потоков предоставляет программисту API для создания и управления потоками. Реализуется двумя способами:

  • полностью в пользовательском пространстве,
  • на уровне ядра при поддержке ОС.

Pthreads

  • Стандарт POSIX (IEEE 1003.1c) — спецификация, а не реализация.
  • Может быть реализована и в user-space, и в kernel-space.
  • Распространена в UNIX-системах: Solaris, Linux, macOS.

Windows threads

  • Часть Windows API.
  • Реализует one-to-one mapping на ядерные потоки.

Java threads

  • Управляются JVM, обычно поверх ядерных потоков ОС.
  • Создаются двумя способами: наследование от Thread или реализация интерфейса Runnable.

10. Неявная многопоточность (Implicit Threading)

Чем больше потоков, тем сложнее обеспечить корректность вручную. Решение — переложить создание и управление потоками с программиста на компиляторы и среды выполнения. Программист описывает задачи, а не сами потоки.

Thread Pools (пулы потоков)

  • Заранее создаётся пул потоков, ожидающих работы.
  • Обслуживание запроса существующим потоком быстрее создания нового.
  • Число потоков ограничено размером пула.
  • Отделение задачи от механизма её исполнения — задачи можно планировать, в т.ч. периодически.
  • Поддерживаются в Windows API.

OpenMP

  • Набор директив компилятора и API для C, C++, FORTRAN.
  • Поддержка параллельного программирования в среде с общей памятью.
  • Программист отмечает параллельные регионы:
#pragma omp parallel
{
    /* создать столько потоков, сколько ядер */
}

#pragma omp parallel for
for (i = 0; i < N; i++) {
    c[i] = a[i] + b[i];     /* цикл выполняется параллельно */
}

Grand Central Dispatch (GCD)

  • Технология Apple для macOS и iOS.
  • Расширения C/C++, API и runtime-библиотека.
  • Блоки кода в синтаксисе ^{ ... }, например: ^{ printf("I am a block"); }
  • Блоки помещаются в очереди и берутся свободными потоками из пула.

Два типа очередей:

  • Serial — FIFO, по одному блоку на удаление; одна на процесс (main queue), плюс пользовательские.
  • Concurrent — FIFO, но несколько блоков могут быть запущены одновременно. Три системных очереди с приоритетами low, default, high.

Другие подходы: Microsoft Threading Building Blocks (TBB), java.util.concurrent.

11. Проблемы многопоточности

Семантика fork() и exec()

  • Должен ли fork() дублировать все потоки процесса, или только вызвавший поток? — В некоторых UNIX-системах есть две версии fork() для каждого варианта.
  • exec() работает как обычно: заменяет процесс целиком, со всеми потоками.

Обработка сигналов

В UNIX сигналы — это способ уведомить процесс о событии. Жизненный цикл сигнала:

  1. Сигнал генерируется событием.
  2. Сигнал доставляется процессу.
  3. Сигнал обрабатывается обработчиком по умолчанию (от ядра) или пользовательским обработчиком.

Куда доставлять сигнал в многопоточном процессе? — Возможны четыре стратегии:

  • в тот поток, к которому относится сигнал;
  • каждому потоку процесса;
  • определённым потокам;
  • выделенному потоку, принимающему все сигналы процесса.

Отмена потока (Thread Cancellation)

Завершение целевого потока до того, как он закончит работу. Два подхода:

ПодходПоведение
AsynchronousПоток немедленно завершается — даже посреди обновления данных.
Deferred (отложенная)Поток получает уведомление и сам периодически проверяет, не пора ли завершиться. Завершение происходит в точке отмены (pthread_testcancel()), затем вызывается cleanup handler.

Тип по умолчанию — deferred. В Linux отмена потоков реализована через сигналы. В большинстве случаев отложенная отмена предпочтительнее асинхронной.

Thread-Local Storage (TLS)

  • Позволяет каждому потоку иметь свою копию данных.
  • Полезно, когда программист не контролирует создание потоков (например, в пуле).
  • Отличается от локальных переменных: те видны только в одном вызове функции, TLS — между вызовами.
  • Похоже на static-данные, но уникально для каждого потока.

Scheduler Activations

  • Модели M:M и Two-level требуют коммуникации между уровнями, чтобы поддерживать правильное число ядерных потоков.
  • Между пользовательскими и ядерными потоками используется промежуточная структура — LWP (Lightweight Process).
  • LWP выглядит как «виртуальный процессор», на котором планируется пользовательский поток.
  • Каждый LWP привязан к ядерному потоку.
  • Upcalls — механизм коммуникации из ядра в библиотеку потоков, позволяющий приложению поддерживать корректное число ядерных потоков.

12. Примеры из ОС

Windows Threads

  • Реализует Windows API — главный API для Win 98/NT/2000/XP/7.
  • Модель one-to-one, kernel-level.
  • Каждый поток содержит: ID, набор регистров, отдельные user-stack и kernel-stack, область приватных данных (для runtime и DLL).
  • Регистры + стеки + приватная область = контекст потока.

Основные структуры данных потока в Windows:

СтруктураСодержимоеГде
ETHREADExecutive thread block; указатель на процесс и на KTHREADKernel space
KTHREADИнформация о планировании и синхронизации, kernel-stack, указатель на TEBKernel space
TEBThread environment block; ID потока, user-stack, TLSUser space

Linux Threads

  • В Linux потоки называются tasks, понятия «процесс» и «поток» не различаются.
  • Создание — через системный вызов clone().
  • clone() позволяет дочерней задаче разделять адресное пространство родителя; поведение настраивается флагами.
  • Структура struct task_struct содержит указатели на структуры данных процесса — они либо разделяются, либо у каждой задачи свои.
Главный вывод: поток — это «лёгкий» процесс, разделяющий ресурсы с другими потоками того же процесса. Многопоточность даёт отзывчивость, экономию и масштабируемость, но требует тщательной работы с синхронизацией, сигналами, отменой и переносимостью между моделями user/kernel-потоков.

1. What is a thread

A thread is the basic unit of CPU utilization. Every thread has:

  • a thread ID,
  • a program counter (PC),
  • a register set,
  • its own stack.
All threads in the same process share: the code section, data section, open files, and signals — i.e. resources owned by the process.

A traditional process has a single thread of control. With multiple threads, it can perform more than one task at a time.

Example. A word processor: one thread for graphics, another for keystrokes, a third for spell/grammar checking in the background.

2. Motivation

  • Most modern applications are multithreaded.
  • Distinct tasks inside one application become separate threads: update display, fetch data, spell-check, answer a network request.
  • Process creation is heavy-weight; thread creation is light-weight.
  • Threading can simplify code and increase efficiency.
  • Kernels themselves are generally multithreaded.

3. Benefits of multithreading

Responsiveness
A process can keep running even if part of it is blocked — critical for user interfaces.
Resource sharing
Threads share process resources by default — easier than shared memory or message passing.
Economy
Cheaper than creating processes; thread switching has lower overhead than context switching.
Scalability
A multithreaded process can take advantage of multiprocessor / multicore architectures.

4. Multicore programming

Multicore and multiprocessor systems put pressure on programmers. The main challenges:

  • Dividing activities — how to split the work.
  • Balance — keep all cores equally busy.
  • Data splitting — which data goes to which thread.
  • Data dependency — synchronization across threads.
  • Testing & debugging — especially hard for concurrent code.

Types of parallelism

TypeWhat is partitionedThreads do…
Data parallelismSubsets of the same data across coresThe same operation on each core
Task parallelismThreads/tasks distributed across coresEach thread runs a unique operation

As the number of threads grows, hardware support grows with it: CPUs have multiple cores, and each core may have multiple hardware threads. Example: Oracle SPARC T4 — 8 cores × 8 hardware threads per core.

5. Concurrency vs parallelism

Concurrency — multiple threads are making progress, not necessarily at the same instant. Achievable even on a single core via the scheduler.

Parallelism — multiple threads are actually executing simultaneously. Requires a multicore system.

Single CPU → concurrency only. True parallelism needs multiple CPUs.

6. Amdahl's Law

Quantifies the performance gain from adding cores to an application that has both serial and parallel parts.

speedup = 1 / ( S + (1 − S) / N )

where S = serial fraction, N = number of cores.

  • If the application is 75% parallel / 25% serial, moving from 1 to 2 cores gives a speedup of 1.6×.
  • As N → ∞, speedup approaches 1 / S.
  • Implication: the serial portion has a disproportionate effect on the gains achievable from more cores.

Open question raised in the lecture: does the law account for contemporary multicore systems? — With many cores, cache hierarchies, inter-core communication, and NUMA effects matter; Amdahl's Law becomes an upper bound, not the full picture.

7. User threads and kernel threads

User threads
Managed by a user-level threads library.
  • POSIX Pthreads
  • Windows threads
  • Java threads
Kernel threads
Supported directly by the kernel.
  • Windows
  • Solaris
  • Linux
  • Tru64 UNIX
  • macOS

8. Multithreading models

User threads must ultimately be mapped to kernel threads to execute on the CPU. There are four common models.

Many-to-One

User T₁User T₂User T₃→Kernel Thread
  • Many user threads mapped to a single kernel thread.
  • One blocking thread blocks all.
  • Cannot run in true parallel on multicore.
  • Examples: Solaris Green Threads, GNU Portable Threads.

One-to-One

User T₁→Kernel T₁User T₂→Kernel T₂
  • Each user thread maps to one kernel thread.
  • More concurrency than M:1.
  • Overhead may cap the number of threads per process.
  • Examples: Windows, Linux, Solaris ≥ 9.

Many-to-Many

User T₁User T₂User T₃→Kernel T₁Kernel T₂
  • Many user threads multiplexed onto several kernel threads.
  • OS creates enough kernel threads for the workload.
  • Examples: Solaris < 9, Windows with ThreadFiber.

Two-level

Like Many-to-Many, but additionally allows a user thread to be bound to a specific kernel thread. Examples: IRIX, HP-UX, Tru64 UNIX, Solaris ≤ 8.

9. Thread libraries

A thread library exposes an API for creating and managing threads. Two implementation strategies:

  • entirely in user space,
  • kernel-level, supported by the OS.

Pthreads

  • POSIX standard (IEEE 1003.1c) — a specification, not an implementation.
  • Can be user- or kernel-level.
  • Common across UNIX: Solaris, Linux, macOS.

Windows threads

  • Part of the Windows API.
  • Uses a one-to-one mapping to kernel threads.

Java threads

  • Managed by the JVM, typically over the host OS's kernel threads.
  • Two creation paths: extend Thread or implement Runnable.

10. Implicit threading

As thread counts grow, getting program correctness right manually becomes harder. The idea: shift thread creation and management from the programmer to compilers and runtime libraries. The programmer identifies tasks, not threads.

Thread Pools

  • A pool of threads is created in advance, waiting for work.
  • Servicing a request with an existing thread is faster than creating a new one.
  • Bounds the total number of threads to the pool size.
  • Decoupling task from thread enables different scheduling strategies (e.g., periodic tasks).
  • Supported by the Windows API.

OpenMP

  • A set of compiler directives plus an API for C, C++, FORTRAN.
  • Targets parallel programming in shared-memory environments.
  • The programmer marks parallel regions:
#pragma omp parallel
{
    /* create as many threads as there are cores */
}

#pragma omp parallel for
for (i = 0; i < N; i++) {
    c[i] = a[i] + b[i];     /* loop runs in parallel */
}

Grand Central Dispatch (GCD)

  • Apple technology for macOS and iOS.
  • Extensions to C/C++, an API, and a runtime library.
  • Code blocks in ^{ ... } syntax: ^{ printf("I am a block"); }
  • Blocks are placed in dispatch queues and picked up by available threads from a pool.

Two queue types:

  • Serial — FIFO, one block at a time; one queue per process (the main queue), plus user-created.
  • Concurrent — FIFO, but several blocks may be removed at a time. Three system-wide queues at priorities low, default, high.

Other approaches: Microsoft Threading Building Blocks (TBB), java.util.concurrent.

11. Threading issues

Semantics of fork() and exec()

  • Does fork() duplicate all threads, or only the calling thread? Some UNIXes provide two versions of fork() — one for each behavior.
  • exec() works as usual: it replaces the entire running process, including all threads.

Signal handling

Signals notify a UNIX process that some event occurred. Lifecycle:

  1. The signal is generated by an event.
  2. The signal is delivered to a process.
  3. The signal is handled by either the default handler (kernel) or a user-defined one.

Where do you deliver a signal in a multithreaded process? Four strategies:

  • to the thread the signal applies to;
  • to every thread in the process;
  • to certain threads only;
  • to one designated thread that receives all signals.

Thread cancellation

Terminating the target thread before it finishes. Two approaches:

ApproachBehavior
AsynchronousTerminates the target immediately — even mid-update.
DeferredThe target is told to terminate; it periodically checks. Cancellation happens at cancellation points (pthread_testcancel()), then a cleanup handler runs.

The default type is deferred. On Linux, cancellation is implemented through signals. In most circumstances, deferred is preferred over asynchronous.

Thread-Local Storage (TLS)

  • Gives each thread its own copy of data.
  • Useful when you don't control thread creation (e.g., when using a pool).
  • Different from local variables: locals live only inside one function call; TLS persists across calls.
  • Similar to static data, but per-thread.

Scheduler activations

  • The M:M and Two-level models need communication to keep the right number of kernel threads allocated.
  • An intermediate structure — the LWP (lightweight process) — sits between user and kernel threads.
  • An LWP looks like a virtual processor onto which the user library schedules user threads.
  • Each LWP is attached to a kernel thread.
  • Upcalls are the kernel-to-thread-library communication that lets the application keep the kernel-thread count correct.

12. OS examples

Windows Threads

  • Implements the Windows API (Win 98, NT, 2000, XP, 7).
  • Uses a one-to-one, kernel-level mapping.
  • Each thread holds: a thread ID, a register set, separate user and kernel stacks, and a private storage area for the runtime/DLLs.
  • The registers + stacks + private storage = the thread's context.

Primary data structures for a Windows thread:

StructureContentsLives in
ETHREADExecutive thread block; pointer to owning process and to KTHREADKernel space
KTHREADScheduling & synchronization info, kernel-mode stack, pointer to TEBKernel space
TEBThread environment block; thread ID, user-mode stack, TLSUser space

Linux Threads

  • Linux calls them tasks and does not distinguish between processes and threads.
  • Created via the clone() system call.
  • clone() lets a child task share the parent's address space; flags control exactly which resources are shared.
  • A struct task_struct points to per-task data structures — shared or unique depending on the flags.
Take-away: a thread is a "lightweight process" that shares resources with sibling threads in the same process. Multithreading delivers responsiveness, economy, and scalability — but comes with a set of subtle hazards (synchronization, signal delivery, cancellation, user/kernel mapping) that the OS, the runtime, and the programmer must each handle correctly.
