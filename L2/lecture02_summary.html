Operating-System Structures

Структура операционных систем — резюме и пояснения

1. Сервисы операционной системы

ОС — это среда выполнения программ. Её сервисы делятся на две группы: помощь пользователю и эффективная работа самой системы.

Сервисы для пользователя

  • Пользовательский интерфейс (UI) — CLI, GUI или batch.
  • Выполнение программ — загрузка в память, запуск, нормальное или аварийное завершение.
  • Операции ввода-вывода (I/O) — работа с файлами и устройствами.
  • Файловая система — создание, чтение, запись, поиск, права доступа.
  • Коммуникации — через разделяемую память или передачу сообщений (пакетов).
  • Обнаружение ошибок — в CPU, памяти, устройствах I/O, программах пользователя.

Сервисы для самой системы

  • Распределение ресурсов — CPU, память, дисковое пространство, устройства.
  • Учёт (Accounting) — сколько и каких ресурсов потребляет каждый пользователь.
  • Защита (Protection) — контроль доступа к ресурсам системы.
  • Безопасность (Security) — аутентификация и защита от внешних угроз.

2. Интерфейс пользователя

CLI

Командная строка

Командный интерпретатор (shell) получает команду и выполняет её. Часто бывает несколько «shell»-ов: bash, zsh, PowerShell. Если команды — это просто имена программ, добавление новых функций не требует модификации shell.

GUI

Графический интерфейс

Метафора «рабочего стола», изобретена в Xerox PARC. Современные ОС обычно совмещают CLI и GUI: Windows, macOS (Aqua + UNIX kernel), Linux (CDE/KDE/GNOME).

Touch

Сенсорный интерфейс

Жесты вместо мыши, виртуальная клавиатура, голосовые команды. Требует нового подхода к проектированию.

3. Системные вызовы (System Calls)

System call — это программный интерфейс к сервисам ОС. Обычно программисты не вызывают их напрямую, а используют API:

  • Win32 API — Windows
  • POSIX API — UNIX, Linux, macOS
  • Java API — для JVM

Как это работает

  1. Каждому системному вызову присвоен номер.
  2. Интерфейс системных вызовов хранит таблицу, индексированную этими номерами.
  3. Интерфейс вызывает нужный сервис в ядре и возвращает статус и результат.
  4. Программист работает только с API — детали реализации скрыты библиотекой времени выполнения.

Способы передачи параметров

СпособОписаниеОсобенности
РегистрыПараметры передаются в регистрах CPUПростой, но ограниченный по числу параметров
Блок в памятиАдрес блока передаётся в регистреLinux, Solaris — без ограничения по длине
СтекПрограмма кладёт параметры в стек, ОС снимаетБез ограничения по длине

4. Типы системных вызовов

Управление процессами

  • create / terminate
  • load / execute
  • get / set attributes
  • wait, signal events
  • allocate / free memory
  • locks для разделяемых данных

Управление файлами

  • create / delete
  • open / close
  • read / write / reposition
  • get / set атрибуты файла

Управление устройствами

  • request / release
  • read / write
  • attach / detach устройств

Поддержка информации

  • get / set time, date
  • системные данные
  • атрибуты процессов и файлов

Коммуникации

  • создание/удаление соединений
  • передача сообщений (client → server)
  • shared-memory модель

Защита

  • контроль доступа к ресурсам
  • get / set permissions
  • allow / deny user access

Примеры систем

MS-DOS — однозадачная. Shell запускается при загрузке, загружает программу в память поверх себя. После выхода программы shell перезагружается. Процессы не создаются.
FreeBSD — многозадачная, UNIX-подобная. Логин запускает shell, который через fork() создаёт процесс и через exec() загружает программу. Код выхода: 0 — успех, >0 — ошибка.

5. Системные программы

Это то, как пользователь обычно «видит» ОС — а не системные вызовы.

  • Работа с файлами: create, delete, copy, rename, print, list.
  • Информация о состоянии: дата, время, память, диск, число пользователей; иногда — реестр.
  • Модификация файлов: текстовые редакторы, поиск, преобразования.
  • Поддержка языков: компиляторы, ассемблеры, отладчики, интерпретаторы.
  • Загрузка и выполнение программ: loader'ы, linkage editors, overlay-loaders.
  • Коммуникации: e-mail, удалённый вход, передача файлов, веб-страницы.
  • Фоновые службы (daemons): запускаются при boot, работают в user context — диск, печать, журналирование.
  • Прикладные программы — не считаются частью ОС.

6. Проектирование и реализация ОС

Задача проектирования ОС не имеет «единственного правильного решения». Начинают с целей:

  • Цели пользователя: удобство, лёгкость освоения, надёжность, безопасность, скорость.
  • Цели системы: простота проектирования, реализации, поддержки; гибкость, надёжность, эффективность.
Главный принцип — разделить политику и механизм:
Mechanism — как что-то делать.
Policy — что именно делать.
Такое разделение даёт максимальную гибкость: политику можно менять без изменения механизма (пример — таймер).

Языки реализации

Раньше — ассемблер, потом Algol/PL/1, сейчас — C/C++. Обычно смешанные: нижние уровни на ассемблере, основная часть на C, утилиты на Perl/Python/shell. Высокоуровневые языки портируются легче, но работают медленнее.

7. Структуры ОС

Простая структура — MS-DOS

Не разделена на модули, интерфейсы и уровни функциональности перемешаны. Цель — максимум функций при минимуме места.

Монолитная структура — UNIX

Состоит из двух частей: системные программы и ядро. Ядро — всё между интерфейсом системных вызовов и аппаратурой. Много функций на одном уровне.

Слоистая (Layered) структура

ОС разделена на уровни (0 — оборудование, N — пользовательский интерфейс). Каждый слой использует только функции нижележащих слоёв. Преимущество — модульность, недостаток — производительность.

Микроядро (Microkernel) — Mach

Минимум функциональности в ядре, остальное — в user space. Коммуникация модулей через message passing.

+ Плюсы− Минусы
Легче расширятьНакладные расходы на user↔kernel коммуникацию
Легче портировать
Надёжнее (меньше кода в kernel mode)
Безопаснее

Модули (Modules)

Современные ОС используют загружаемые модули ядра (Linux, Solaris). Объектно-ориентированный подход: компоненты общаются через известные интерфейсы и подгружаются по необходимости.

8. Гибридные системы

На практике чистых моделей не существует — современные ОС комбинируют подходы.

  • Linux / Solaris — монолитное ядро + модульная загрузка.
  • Windows — преимущественно монолитная + микроядро для подсистем.
  • Mac OS X — гибрид: микроядро Mach + BSD UNIX + I/O Kit + загружаемые kernel extensions; сверху Aqua UI и Cocoa.
  • iOS — на базе macOS, ARM, Cocoa Touch (Objective-C), Core Services (облако, БД).
  • Android — на базе модифицированного Linux. Приложения на Java компилируются в bytecode и исполняются в Dalvik VM. Open Source, разработан Open Handset Alliance.

9. Отладка и тюнинг

  • Логи с информацией об ошибках.
  • Core dump — снимок памяти упавшего процесса.
  • Crash dump — снимок памяти ядра при падении ОС.
  • Profiling — периодическая выборка instruction pointer для статистического анализа.
  • DTrace (Solaris, FreeBSD, macOS) — живая инструментовка production-систем через probes.
Закон Кернигана: «Отладка вдвое сложнее, чем написание кода. Поэтому если вы пишете код настолько умно, насколько способны, то по определению вы недостаточно умны, чтобы его отладить.»

10. Генерация и загрузка системы

SYSGEN

Программа, которая собирает информацию о конкретной конфигурации оборудования и собирает специализированное ядро — оно эффективнее, чем универсальное.

Загрузка системы (Boot)

  1. При включении CPU начинает выполнение с фиксированного адреса.
  2. В ROM/EEPROM хранится начальный boot-код — bootstrap loader.
  3. Bootstrap loader находит ядро, загружает его в память и запускает.
  4. Иногда — двухступенчатая загрузка: boot block с фиксированного места → bootstrap loader с диска.
  5. GRUB — популярный загрузчик, позволяет выбирать ядро из нескольких дисков/версий/опций.

Главные выводы

  • ОС предоставляет сервисы двух категорий: для пользователя и для самой себя.
  • Доступ к сервисам — через системные вызовы, обычно скрытые за API.
  • Главный принцип проектирования: отделять политику от механизма.
  • Структуры: простая → монолитная → слоистая → микроядро → модульная → гибридная.
  • Современные ОС — почти всегда гибридные.
  • Загрузка идёт через bootstrap loader из ROM.

1. Operating System Services

An OS provides an environment for executing programs. Its services fall into two groups: helping the user and running the system itself efficiently.

Services for the user

  • User Interface (UI) — CLI, GUI, or batch.
  • Program execution — load into memory, run, end normally or abnormally.
  • I/O operations — with files or devices.
  • File-system manipulation — read/write, create/delete, search, permissions.
  • Communications — shared memory or message passing across the network.
  • Error detection — in CPU, memory, devices, user programs.

Services for the system itself

  • Resource allocation — CPU cycles, RAM, disk, I/O devices.
  • Accounting — track who consumes what resources.
  • Protection — controlled access to system resources.
  • Security — authentication and external threat defense.

2. User Interface

CLI

Command Line

The command interpreter (shell) fetches a command and executes it. Often multiple flavors exist (bash, zsh, PowerShell). When commands are just program names, new features don't require shell modification.

GUI

Graphical UI

The desktop metaphor, invented at Xerox PARC. Modern systems combine CLI and GUI: Windows, macOS (Aqua + UNIX kernel), Linux (CDE/KDE/GNOME).

Touch

Touchscreen

Gestures instead of mouse, virtual keyboard, voice commands. Requires fresh UX design.

3. System Calls

A system call is the programming interface to OS services. Programmers rarely call them directly — they use an API:

  • Win32 API — Windows
  • POSIX API — UNIX, Linux, macOS
  • Java API — JVM

How it works

  1. Each system call has a number.
  2. The system-call interface keeps a table indexed by these numbers.
  3. The interface invokes the call in kernel and returns status and values.
  4. The programmer only sees the API — implementation details are hidden by the run-time library.

Parameter passing

MethodDescriptionNotes
RegistersParameters in CPU registersSimplest, limited by register count
Memory blockAddress passed in a registerUsed by Linux and Solaris; no length limit
StackProgram pushes, OS popsNo length limit

4. Types of System Calls

Process control

  • create / terminate
  • load / execute
  • get / set attributes
  • wait, signal events
  • allocate / free memory
  • locks for shared data

File management

  • create / delete
  • open / close
  • read / write / reposition
  • get / set file attributes

Device management

  • request / release
  • read / write
  • logically attach / detach

Information maintenance

  • get / set time, date
  • system data
  • process / file attributes

Communications

  • create / delete connections
  • message passing (client → server)
  • shared-memory model

Protection

  • access control
  • get / set permissions
  • allow / deny user access

System examples

MS-DOS — single-tasking. Shell loads at boot, loads a program into memory overwriting itself. On program exit, the shell is reloaded. No processes are created.
FreeBSD — multitasking, UNIX-like. Login invokes a shell; fork() creates a process; exec() loads the program. Exit codes: 0 = success, >0 = error.

5. System Programs

Most users' view of the OS is shaped by system programs — not by system calls.

  • File manipulation — create, delete, copy, rename, print, list.
  • Status info — date, time, memory, disk, users; sometimes a registry.
  • File modification — text editors, search, transformations.
  • Language support — compilers, assemblers, debuggers, interpreters.
  • Loading and execution — loaders, linkage editors, overlays.
  • Communications — e-mail, remote login, file transfer, web.
  • Background services / daemons — start at boot, run in user context (disk check, printing, logging).
  • Application programs — not part of the OS.

6. OS Design and Implementation

OS design has no single solution. Start with goals:

  • User goals: convenient, easy to learn, reliable, safe, fast.
  • System goals: easy to design, implement, maintain; flexible, reliable, error-free, efficient.
Key principle — separate policy from mechanism:
Mechanism — how to do something.
Policy — what to do.
The separation maximizes flexibility — policies can change without touching the mechanism (timer is a classic example).

Implementation languages

Early: assembly. Then Algol / PL/1. Now C and C++, usually mixed: assembly at the lowest level, C in the main body, scripting languages (Perl, Python, shell) for utilities. Higher-level code is more portable but slower.

7. OS Structures

Simple — MS-DOS

Not divided into modules; interfaces and functional levels are not well-separated. Goal: maximum functionality in minimum space.

Monolithic — UNIX

Two parts: system programs and the kernel. Kernel = everything between the system-call interface and the hardware — file system, CPU scheduling, memory management — all on one level.

Layered

OS split into layers (level 0 = hardware, level N = user interface). Each layer uses only services from lower layers. Pros: modularity. Cons: performance cost.

Microkernel — Mach

Move as much as possible out of the kernel into user space. Modules communicate via message passing.

+ Pros− Cons
Easier to extendOverhead of user↔kernel communication
Easier to port
More reliable (less kernel-mode code)
More secure

Modules

Modern systems (Linux, Solaris) use loadable kernel modules. Object-oriented in spirit: separate components talk over known interfaces and load on demand.

8. Hybrid Systems

No real-world OS is a pure model — modern systems mix approaches.

  • Linux / Solaris — monolithic kernel + dynamic module loading.
  • Windows — mostly monolithic + microkernel for subsystem personalities.
  • Mac OS X — hybrid: Mach microkernel + BSD UNIX + I/O Kit + kernel extensions; Aqua UI and Cocoa on top.
  • iOS — based on macOS, ARM CPU, Cocoa Touch (Objective-C), Core Services for cloud and databases.
  • Android — modified Linux kernel. Java apps compile to bytecode, run on Dalvik VM. Open Source, Open Handset Alliance.

9. Debugging and Performance Tuning

  • Log files with error information.
  • Core dump — memory snapshot of a crashed process.
  • Crash dump — kernel memory snapshot on OS failure.
  • Profiling — periodic sampling of the instruction pointer for statistical trends.
  • DTrace (Solaris, FreeBSD, macOS) — live instrumentation of production systems via probes.
Kernighan's Law: "Debugging is twice as hard as writing the code in the first place. Therefore, if you write the code as cleverly as possible, you are, by definition, not smart enough to debug it."

10. System Generation and Boot

SYSGEN

A program that gathers info about the specific hardware configuration and produces a system-tuned kernel — more efficient than a generic one.

System Boot

  1. Power-on: CPU starts executing at a fixed memory address.
  2. ROM/EEPROM holds the initial boot code — the bootstrap loader.
  3. The bootstrap loader locates the kernel, loads it into memory, and starts it.
  4. Sometimes two-step: a boot block at a fixed location loaded by ROM code, which loads a bootstrap loader from disk.
  5. GRUB — popular bootloader; supports kernel selection across disks, versions, and options.

Key Takeaways

  • The OS provides two categories of services: for users and for itself.
  • Services are accessed via system calls, usually hidden behind an API.
  • Core design principle: separate policy from mechanism.
  • Structures: simple → monolithic → layered → microkernel → modular → hybrid.
  • Real modern systems are almost always hybrid.
  • Booting begins in ROM with a bootstrap loader.
