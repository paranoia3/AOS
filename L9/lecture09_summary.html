Advanced Operating Systems · Lecture 9

Security

Безопасность — резюме и пояснения

Эта лекция охватывает безопасность вычислительных систем: чем отличается security от protection, какие категории нарушений (конфиденциальность, целостность, доступность, кража сервиса, DoS), какие методы атак (masquerading, replay, MITM, hijacking, privilege escalation), четырёхуровневая модель защиты, угрозы программам (Trojan, malware, ransomware, переполнение буфера, виды вирусов), сетевые угрозы (worms, port scanning, DoS/DDoS), криптография как инструмент (симметричное и асимметричное шифрование, RSA, MAC и цифровая подпись, TLS), аутентификация пользователей (пароли, salt, OTP, биометрия, MFA), защитные меры (defense in depth, IDS, sandbox, firewall), классификация D/C/B/A и пример Windows 10.

1. Security vs Protection

Security — мера уверенности в том, что целостность системы и её данных будет сохранена.

Protection — набор механизмов, контролирующих доступ процессов и пользователей к ресурсам.

  • Protection — внутренняя проблема ОС.
  • Security учитывает и систему, и окружение: людей, здания, бизнес, ценности, угрозы.
  • Абсолютной защиты не бывает; цель — сделать стоимость атаки достаточно высокой, чтобы отпугнуть большинство.
  • Защититься от случайных потерь проще, чем от злонамеренных.

2. Проблема безопасности и категории нарушений

Система безопасна, если ресурсы используются и доступны так, как задумано, при любых обстоятельствах — что, по сути, недостижимо.

  • Threat — потенциальное нарушение безопасности.
  • Attack — попытка нарушения; случайная или злонамеренная.
  • Intruders / crackers — те, кто пытается взломать защиту.

Категории нарушений

КатегорияЧто произошло
Breach of confidentialityНесанкционированное чтение данных.
Breach of integrityНесанкционированная модификация данных.
Breach of availabilityНесанкционированное уничтожение данных.
Theft of serviceНесанкционированное использование ресурсов.
Denial of Service (DoS)Препятствование легитимному использованию.

3. Методы атак

  • Masquerading — выдать себя за авторизованного пользователя и эскалировать привилегии.
  • Replay attack — повторное воспроизведение перехваченного сообщения, как есть или с модификацией.
  • Man-in-the-middle (MITM) — атакующий встроен в поток данных, прикидываясь и отправителем для получателя, и наоборот.
  • Session hijacking — перехват уже установленной сессии, минуя аутентификацию.
  • Privilege escalation — доступ выходит за рамки, что положено пользователю или ресурсу.

4. Четыре уровня безопасности

Безопасность должна работать на всех четырёх уровнях:

Physical
Дата-центры, серверы, подключённые терминалы.
Application
Безобидные и вредоносные приложения.
Operating System
Механизмы защиты, отладки.
Network
Перехват, разрыв связи, DoS.

Безопасность настолько слаба, насколько слабо самое слабое звено. Люди — тоже риск (фишинг, социальная инженерия). При этом слишком много безопасности делает систему неюзабельной.

5. Угрозы программам

Trojan Horse

Сегмент кода, который злоупотребляет своим окружением. Эксплуатирует механизмы, позволяющие чужому коду выполняться от имени других пользователей. Включает spyware, всплывающие окна, скрытые каналы. До 80% спама доставляется системами, заражёнными spyware.

Trap Door (бэкдор)

Специальный идентификатор пользователя или пароль, обходящий обычную аутентификацию. Может быть встроен даже в компилятор. Сложен в обнаружении.

Malware

  • Trojan Horse — программа, действующая скрытно.
  • Spyware — часто ставится вместе с легальным ПО, показывает рекламу, собирает данные.
  • Ransomware — шифрует данные, требует выкуп за расшифровку.
  • Также: trap doors, logic bombs.
  • Все нарушают принцип наименьших привилегий (Least Privilege).
  • Цель часто — оставить Remote Access Tool (RAT) для повторного доступа.

6. Переполнение буфера и инъекция кода

Классический пример небезопасного C-кода:

#include <stdio.h>
#define BUFFER_SIZE 256
int main(int argc, char *argv[])
{
    char buffer[BUFFER_SIZE];
    if (argc < 2)
        return -1;
    else {
        strcpy(buffer, argv[1]);   // нет проверки длины
        return 0;
    }
}

Code Injection

  • Возникает, когда системный код не злонамеренный, но имеет баги, позволяющие добавлять/менять исполняемый код.
  • Следствие плохих практик в низкоуровневых языках (C/C++) с прямым доступом к памяти через указатели.
  • Цель — buffer overflow: код подкладывают в буфер и заставляют исполниться.
  • Script kiddies используют готовые эксплоиты, не понимая внутренностей.
  • Часто используется trampoline — переход на чужой код.

Возможные последствия: shell с правами процесса-жертвы, открытие сетевого порта, удаление файлов, загрузка вредоноса. Атака может пройти через сеть, обходя firewall на разрешённых соединениях.

Защиты:

  • отключить выполнение со стека (NX-бит / non-executable);
  • добавить бит «non-executable» в page table (SPARC, x86);
  • code review — программисты вычитывают код друг друга на предмет логических ошибок.

7. Вирусы

  • Фрагмент кода, встроенный в легитимную программу.
  • Самораспространяется, нацелен на заражение других машин.
  • Очень специфичен к CPU-архитектуре, ОС и приложению.
  • Распространяется обычно через email или как макрос.
  • Virus dropper внедряет вирус в систему.

Категории вирусов

  • File / parasitic,
  • Boot / memory,
  • Macro,
  • Source code,
  • Polymorphic — каждый раз другая «сигнатура»,
  • Encrypted, Stealth, Tunneling, Multipartite, Armored.

Почему чаще всего атакуют Windows? Самая массовая, по умолчанию все — администраторы, монокультура «вредна». Атаки давно перешли из «научных экспериментов» в инструмент организованной преступности: ботнеты для спама и DDoS, кейлоггеры для паролей и кредиток.

8. Сетевые угрозы

Worms (черви)

Самостоятельные программы, использующие механизм спавна. Знаменитый Internet worm:

  • Использовал UNIX-сетевые возможности и баги в finger и sendmail.
  • Эксплуатировал trust relationships у rsh — доступ без пароля.
  • Grappling hook — 99 строк C, загружал основной код червя.
  • Подобрав один хост, пытался заразить связанные системы.
  • Брутфорсил пароли других пользователей локально.
  • Если цель уже заражена — обычно прекращал, но каждый 7-й раз всё равно заражал (защита от фейковых «уже заражено»).

Port Scanning

  • Автоматический перебор портов на диапазоне IP-адресов.
  • Определение протокола сервиса, ОС, версии.
  • nmap — сканирует все порты диапазона; nessus — база протоколов, багов и эксплоитов.
  • Часто запускается с zombie-систем для маскировки.

Denial of Service

  • Перегружают цель, чтобы она не могла обслуживать.
  • DDoS — Distributed DoS, атака с многих машин одновременно.
  • Пример: рукопожатие IP (SYN flood) — у ОС лимит на полуоткрытые соединения.
  • Как отличить DDoS от того, что сайт просто популярный? — Непросто.
  • Может быть случайным (студенты, сломавшие fork()) или намеренным (вымогательство, наказание).

9. Криптография: шифрование

Криптография — самый широкий инструмент безопасности. На локальной машине ОС сама управляет PID и портами; в сети источник и получатель нельзя доверять без криптографии — IP-адрес ничего не гарантирует.

Криптография ограничивает множество потенциальных отправителей и/или получателей сообщения и основана на секретах (ключах).

Шифрование (Encryption)

Ограничивает множество получателей. Алгоритм состоит из:

  • K — множество ключей, M — сообщений, C — шифротекстов;
  • E : K → (M → C) и D : K → (C → M);
  • обе функции должны быть эффективно вычисляемы.

Ключевое свойство: вычислить m из c возможно, только если есть ключ k. Шифротекст часто открыт (в сети), поэтому невыводимость k из c критически важна.

Симметричное шифрование

  • Один и тот же ключ шифрует и расшифровывает → ключ должен быть секретным.
  • DES — был стандартом, теперь небезопасен (короткий ключ).
  • Triple-DES — DES трижды с 2–3 ключами.
  • AES (2001, NIST) — блочный шифр, ключи 128/192/256 бит, блоки 128 бит.
  • RC4 — потоковый шифр (Wi-Fi), известны уязвимости.

Асимметричное шифрование (public-key)

  • У каждого пользователя две ключа:
    • public — публикуется, используется для шифрования;
    • private — известен только владельцу, используется для расшифровки.
  • RSA — самый распространённый. Основан на том, что:
    • проверять простоту числа — легко;
    • раскладывать число на простые множители — неизвестно как делать эффективно.

Формально:

N = p · q (p, q — большие простые)
Eₖₑ,N(m) = m^kₑ mod N
Dₖd,N(c) = c^kd mod N
kₑ · kd ≡ 1 (mod (p−1)(q−1))

Игрушечный пример: p = 7, q = 13 → N = 91, (p−1)(q−1) = 72. Выбираем kₑ = 5, тогда kd = 29. Публичный ключ (5, 91), приватный (29, 91). Сообщение 69 → шифротекст 62 → обратно 69.

Замечания:

  • Симметричная криптография основана на трансформациях, асимметричная — на математических функциях.
  • Асимметричная гораздо более вычислительно дорогая → обычно не используется для шифрования больших объёмов.

10. Аутентификация сообщений: hash, MAC, цифровая подпись

Authentication

Ограничивает множество отправителей сообщения, дополняет шифрование. Также доказывает, что сообщение не было изменено.

Алгоритм:

  • S : K → (M → A) — генерация аутентификатора;
  • V : K → (M × A → {true, false}) — верификация.

Если Vk(m, a) = true, значит сообщение не модифицировано и отправитель имеет k.

Hash-функции

  • Создают маленький блок фиксированной длины — message digest.
  • Должны быть collision-resistant: невозможно найти m' ≠ m с H(m) = H(m').
  • MD5 (128 бит), SHA-1 (160 бит).
  • Сами по себе не годятся как аутентификаторы: если злодей подменит m на m' и пересчитает H(m') — никто не заметит. Поэтому H(m) сам надо аутентифицировать.

MAC (Message Authentication Code)

  • Криптографическая контрольная сумма, создаваемая с секретным ключом.
  • Симметричное шифрование, используется для коротких значений.
  • Если МАС применить к H(m) — получим аутентификацию длинных сообщений.
  • Ключ нужен и для генерации, и для проверки → кто может проверить, тот может и подделать.

Цифровая подпись (Digital Signature)

  • На асимметричных ключах. Аутентификатор = цифровая подпись.
  • Любой может проверить подлинность — публичным ключом.
  • ks (signing) — приватный, kv (verify) — публичный, вывести ks из kv невозможно.
  • RSA digital signature: Sks(m) = H(m)^ks mod N; верификация: a^kv mod N = H(m).

Зачем аутентификация, если есть шифрование?

  • Меньше вычислений (кроме RSA-подписей).
  • Аутентификатор обычно короче сообщения.
  • Иногда нужна аутентификация без конфиденциальности (подписанные патчи).
  • База для non-repudiation — отправитель не сможет отрицать факт отправки.

11. Распределение ключей и сертификаты

  • Доставка симметричного ключа — большая проблема, часто решается out-of-band.
  • Асимметричных ключей много — хранят в key ring.
  • Даже распределение публичных ключей не безопасно — возможна MITM-атака на асимметричную криптографию.

Цифровые сертификаты

  • Доказательство кому принадлежит публичный ключ.
  • Публичный ключ цифрово подписан доверенной стороной.
  • Certificate Authority (CA) — доверенная сторона; их публичные ключи входят в дистрибутивы браузеров.
  • CA подписывают ключи других CA — образуется цепочка доверия.

12. TLS и реализация криптографии

Криптографию можно встроить на разных уровнях модели OSI:

  • SSL/TLS — на транспортном уровне;
  • IPSec — на сетевом, основа VPN; IKE — обмен ключами.

Почему не только на нижнем уровне? Иногда нужны знания, доступные только сверху (аутентификация пользователя, доставка email).

TLS / SSL

  • Криптографический протокол, ограничивающий обмен сообщениями двумя компьютерами.
  • Браузер ↔ web-server, защита кредиток.
  • Сервер верифицируется сертификатом.
  • Асимметричная криптография используется, чтобы обменяться симметричным session key, а основной трафик — уже симметрично (быстрее).

13. Аутентификация пользователей

Правильное определение пользователя критично — на этом строится вся защита. Чаще всего — пароли.

Правила обращения с паролями

  • Хранить в секрете, часто менять, не повторяться.
  • Не позволять «угадываемых» паролей.
  • Логировать неудачные попытки (но не сами пароли!).
  • Хранить только в зашифрованном виде или использовать одноразовые.

Угрозы паролям

  • Sniffing (перехват сети) — лечится шифрованием.
  • Shoulder surfing — подглядывание.
  • Trojan keystroke logger.

Хранение паролей

  • Хранится только хэш (UNIX: /etc/shadow, доступен только суперпользователю).
  • Хэш-функция: легко вычислить, трудно обратить.
  • Salt — добавляется к паролю, чтобы одинаковые пароли давали разные хэши; защита от rainbow-таблиц.

One-time passwords (OTP)

Пароль вычисляется по функции и seed одновременно у пользователя и на сервере; железный токен / калькулятор / key fob. Часто меняется → replay невозможен.

Биометрия

Физический атрибут: отпечаток пальца, скан ладони…

Multi-Factor Authentication (MFA)

Два или более фактора, например: USB-донгл + биометрия + пароль.

14. Защитные меры

Defense in depth — основной принцип: несколько уровней защиты.

  • Security policy — что именно защищаем.
  • Vulnerability assessment — сравнение реального состояния с политикой.
  • Intrusion Detection (IDS):
    • Signature-based — известные плохие паттерны.
    • Anomaly detection — отклонения от нормы; ловит даже zero-day.
    • Проблемы: false positives и false negatives.
  • Virus protection — поиск сигнатур или запуск в sandbox.
  • Auditing, accounting, logging — журналирование событий.
  • Safe computing — избегать сомнительных источников.

Сводка лучших практик

  • Обучать пользователей: не подключать чужие устройства, не делиться паролями, остерегаться социальной инженерии, понимать, что email — не приватное общение.
  • Защита от фишинга: не открывать ссылки/вложения; подтверждать запросы по телефону.
  • Защищённые коммуникации, физическая защита железа.
  • Минимизировать attack surface — выключить лишние сервисы.
  • Свежее железо/софт, регулярные патчи, только trusted-источники (с code signing).
  • Логи + автоматизированные alerts.
  • Антивирус, IDS, firewall.
  • Сильные пароли, не записывать их где попало.
  • Шифровать накопители и важные файлы.
  • Иметь security policy и поддерживать её актуальной.

15. Firewall и классификация безопасности

Firewall

  • Стоит между доверенными и недоверенными хостами.
  • Ограничивает сетевой доступ между двумя security-доменами.
  • Может быть обойдён: tunneling (например, telnet внутри HTTP), spoofing (правила по IP/имени, которые подделываются).
  • Personal firewall — программный слой на конкретном хосте.
  • Application proxy firewall — понимает прикладной протокол (например, SMTP).
  • System-call firewall — следит за системными вызовами и применяет правила к каждой программе.

Computer Security Classifications (DoD)

Министерство обороны США выделяет четыре дивизиона: A, B, C, D.

КлассЧто обеспечивает
DМинимальная защита.
CDiscretionary protection через аудит.
C1 — кооперирующие пользователи с одним уровнем защиты.
C2 — user-level access control.
BВсе свойства C + каждому объекту — уникальные sensitivity labels. Уровни B1, B2, B3.
AФормальные методы проектирования и верификации безопасности.

16. Пример: Windows 10

  • Безопасность основана на учётных записях: у каждого пользователя уникальный Security ID.
  • Login создаёт security access token: ID пользователя, его групп, специальные привилегии.
  • Каждый процесс получает копию токена. Система проверяет токен при доступе.
  • Используется subject model: subject отслеживает разрешения для каждой программы, запущенной пользователем.
  • У каждого объекта Windows — security descriptor: owner ID, group SID, discretionary ACL, system ACL.
  • Windows ввёл mandatory integrity controls — integrity-метки на каждый защищаемый объект и субъект.
  • Объекты бывают container (например, каталог ФС) и noncontainer; по умолчанию объект в контейнере наследует права от родителя.
  • Проблемы Windows 10: слабые настройки по умолчанию, много сервисов в коробке, много установленных приложений.
Главный вывод: абсолютной безопасности не существует. Цель — поднять стоимость атаки выше потенциальной выгоды. Это требует defense in depth: физическая защита, безопасные приложения, аккуратная ОС, защищённая сеть; криптография для конфиденциальности и аутентификации; продуманная аутентификация пользователей (пароли + соль, OTP, биометрия, MFA); IDS, антивирус, sandbox, firewall; и понимание, что самое слабое звено — обычно человек.

This lecture covers computer-system security: how security differs from protection, the categories of violations (confidentiality, integrity, availability, theft of service, DoS), the attack methods (masquerading, replay, MITM, hijacking, privilege escalation), the four-level model of defense, program threats (Trojans, malware, ransomware, buffer overflow, virus families), network threats (worms, port scanning, DoS/DDoS), cryptography as a security tool (symmetric and asymmetric encryption, RSA, MAC and digital signatures, TLS), user authentication (passwords, salt, OTP, biometrics, MFA), defenses (defense in depth, IDS, sandbox, firewall), the D/C/B/A classification, and the Windows 10 example.

1. Security vs protection

Security is a measure of confidence that the integrity of a system and its data will be preserved.

Protection is the set of mechanisms that control access of processes and users to resources.

  • Protection is an internal problem for the OS.
  • Security considers both the system and its environment: people, buildings, businesses, valuables, threats.
  • Absolute protection is impossible; the goal is to make the cost of an attack high enough to deter most attempts.
  • Defending against accidental losses is easier than against malicious ones.

2. The security problem and violation categories

A system is secure if its resources are accessed and used exactly as intended under all circumstances — which is essentially unachievable.

  • Threat — a potential violation.
  • Attack — an attempted violation; accidental or malicious.
  • Intruders / crackers — those trying to breach security.

Violation categories

CategoryWhat happens
Breach of confidentialityUnauthorized reading of data.
Breach of integrityUnauthorized modification of data.
Breach of availabilityUnauthorized destruction of data.
Theft of serviceUnauthorized use of resources.
Denial of Service (DoS)Prevention of legitimate use.

3. Attack methods

  • Masquerading — pretend to be an authorized user to escalate privileges.
  • Replay attack — replay a captured message, as-is or modified.
  • Man-in-the-middle (MITM) — attacker sits in the data flow, impersonating both sender and receiver.
  • Session hijacking — take over an already-established session, bypassing authentication.
  • Privilege escalation — access beyond what is granted to the user/resource.

4. The four security levels

To be effective, security must operate at all four levels:

Physical
Data centers, servers, attached terminals.
Application
Benign or malicious apps can cause issues.
Operating System
Protection mechanisms, debugging.
Network
Interception, interruption, DoS.

Security is as strong as its weakest link. Humans are a risk too — phishing and social engineering. And conversely, too much security hurts usability.

5. Program threats

Trojan Horse

A code segment that misuses its environment by exploiting mechanisms that let user-written code execute on behalf of others. Includes spyware, pop-ups, covert channels. Up to 80% of spam is delivered by spyware-infected systems.

Trap Door (backdoor)

A specific user ID or password that bypasses normal authentication. Can be embedded even in a compiler. Hard to detect.

Malware

  • Trojan Horse — operates clandestinely.
  • Spyware — often bundled with legitimate software; shows ads, captures data.
  • Ransomware — encrypts data and demands payment.
  • Also: trap doors, logic bombs.
  • All violate the principle of least privilege.
  • Goal is often to leave behind a Remote Access Tool (RAT).

6. Buffer overflow and code injection

The classic insecure C example:

#include <stdio.h>
#define BUFFER_SIZE 256
int main(int argc, char *argv[])
{
    char buffer[BUFFER_SIZE];
    if (argc < 2)
        return -1;
    else {
        strcpy(buffer, argv[1]);   // no length check
        return 0;
    }
}

Code injection

  • Happens when system code is not malicious but has bugs allowing executable code to be added or modified.
  • Result of unsafe practices in low-level languages (C/C++) with direct pointer access.
  • Goal: a buffer overflow that injects code into a buffer and gets it executed.
  • Script kiddies run pre-written exploits without understanding the internals.
  • A trampoline is often used to redirect execution.

Possible outcomes: a shell with the victim process's privileges, an open network port, file deletion, malware download. The attack can traverse a firewall via allowed connections.

Defenses:

  • disable stack execution (NX bit / non-executable);
  • add a "non-executable" bit in the page table (SPARC, x86);
  • code review — programmers read each other's code for flaws.

7. Viruses

  • A code fragment embedded in a legitimate program.
  • Self-replicating, aimed at infecting other computers.
  • Highly specific to CPU architecture, OS, and applications.
  • Usually delivered via email or as a macro.
  • A virus dropper plants the virus on the system.

Virus families

  • File / parasitic,
  • Boot / memory,
  • Macro,
  • Source code,
  • Polymorphic — changes signature each time,
  • Encrypted, Stealth, Tunneling, Multipartite, Armored.

Why is Windows the main target? It's the most common, everyone runs as administrator by default, monoculture is risky. Attacks have moved from "science experiments" to tools of organized crime: botnets for spam and DDoS, keyloggers for passwords and credit cards.

8. Network threats

Worms

Standalone programs that use a spawn mechanism. The famous Internet worm:

  • Exploited UNIX networking features and bugs in finger and sendmail.
  • Abused rsh trust relationships — passwordless access.
  • A grappling hook of 99 lines of C uploaded the main worm.
  • Tried to attack connected systems and to crack local passwords.
  • If the target was already infected, usually stopped — but every 7th time infected it anyway (counter to fake "already infected" claims).

Port scanning

  • Automated probing of ports across IP ranges.
  • Detects which service speaks the protocol, the OS, and version.
  • nmap — scans whole ranges; nessus — has a database of protocols, bugs, and exploits.
  • Often launched from zombie systems to decrease traceability.

Denial of Service

  • Overloads the target so it can't do useful work.
  • DDoS — Distributed DoS, from many sites at once.
  • Example: SYN flood — the OS has a finite half-open connection table.
  • How to distinguish DDoS from a popular website? Hard.
  • Can be accidental (a CS student's bad fork()) or deliberate (extortion, punishment).

9. Cryptography: encryption

Cryptography is the broadest security tool. Locally the OS controls PIDs and ports; over the network, source and destination cannot be trusted without cryptography — IP address proves nothing.

Cryptography constrains the set of potential senders and/or receivers of a message, based on secrets (keys).

Encryption

Constrains receivers. The algorithm has:

  • K — keys, M — messages, C — ciphertexts;
  • E : K → (M → C) and D : K → (C → M);
  • both functions must be efficiently computable.

Essential property: recovering m from c is feasible only with the key k. Because ciphertext is usually exposed (on the wire), it must be infeasible to derive k from it.

Symmetric encryption

  • The same key encrypts and decrypts → the key must be kept secret.
  • DES — once the standard, now insecure (short key).
  • Triple-DES — DES applied three times with 2–3 keys.
  • AES (NIST, 2001) — block cipher, 128/192/256-bit keys, 128-bit blocks.
  • RC4 — stream cipher (e.g., Wi-Fi); known weaknesses.

Asymmetric (public-key) encryption

  • Every user has two keys:
    • public — published, used to encrypt;
    • private — known only to the owner, used to decrypt.
  • RSA is the most common. Built on:
    • testing primality is easy;
    • factoring a number into primes has no known efficient algorithm.

Formally:

N = p · q (p, q — large primes)
Eₖₑ,N(m) = m^kₑ mod N
Dₖd,N(c) = c^kd mod N
kₑ · kd ≡ 1 (mod (p−1)(q−1))

Toy example: p = 7, q = 13 → N = 91, (p−1)(q−1) = 72. Pick kₑ = 5, get kd = 29. Public key (5, 91), private (29, 91). Message 69 → cipher 62 → back to 69.

Notes:

  • Symmetric crypto is based on transformations; asymmetric is based on mathematical functions.
  • Asymmetric is far more compute-intensive → typically not used for bulk data encryption.

10. Message authentication: hash, MAC, digital signatures

Authentication

Constrains the set of senders, complementing encryption. Also proves the message wasn't modified.

Algorithm:

  • S : K → (M → A) — generates an authenticator;
  • V : K → (M × A → {true, false}) — verifies it.

If Vk(m, a) = true, then m wasn't modified and the sender holds k.

Hash functions

  • Produce a small fixed-size message digest.
  • Must be collision-resistant: no feasible m' ≠ m with H(m) = H(m').
  • MD5 (128 bits), SHA-1 (160 bits).
  • By themselves not authenticators: an attacker who swaps m for m' can recompute H(m') undetectably. H(m) itself must be authenticated.

MAC (Message Authentication Code)

  • A cryptographic checksum computed with a secret key.
  • Symmetric scheme; used for short values.
  • Applying a MAC to H(m) gives authentication of long messages.
  • The same key generates and verifies → whoever can verify can also forge.

Digital signatures

  • Built on asymmetric keys; the authenticator is a digital signature.
  • Anyone can verify authenticity using the public key.
  • ks (signing) is private, kv (verify) is public; deriving ks from kv is infeasible.
  • RSA signature: Sks(m) = H(m)^ks mod N; verify: a^kv mod N = H(m).

Why authentication if encryption already exists?

  • Fewer computations (except RSA signatures).
  • Authenticator is usually shorter than the message.
  • Sometimes you want authentication without confidentiality (signed patches).
  • A basis for non-repudiation.

11. Key distribution and certificates

  • Delivering a symmetric key is a huge challenge — often done out-of-band.
  • Asymmetric keys proliferate — kept in a key ring.
  • Even public-key distribution isn't safe — there is an MITM attack on asymmetric crypto.

Digital certificates

  • Proof of who owns a given public key.
  • A public key is digitally signed by a trusted party.
  • Certificate Authorities (CAs) are trusted parties; their public keys ship with browsers.
  • CAs vouch for other CAs by signing their keys — forming a chain of trust.

12. TLS and cryptography implementation

Cryptography can sit at various layers of the OSI model:

  • SSL/TLS — at the transport layer;
  • IPSec — at the network layer, the basis of VPNs; IKE handles key exchange.

Why not always at the lowest layer? Some logic requires knowledge only available higher up — user authentication, email delivery.

TLS / SSL

  • Cryptographic protocol that lets two computers exchange messages only with each other.
  • Browser ↔ web server; secure credit-card transmission.
  • The server is verified by a certificate.
  • Asymmetric crypto is used to establish a symmetric session key; bulk traffic uses the symmetric key (much faster).

13. User authentication

Identifying the user correctly is critical — protection depends on user IDs. Usually done via passwords.

Password rules

  • Kept secret, rotated, no repeats.
  • No "guessable" passwords.
  • Log invalid attempts (but not the passwords themselves!).
  • Store encrypted or use one-time passwords.

Threats to passwords

  • Sniffing (network) — encryption helps.
  • Shoulder surfing.
  • Trojan keystroke loggers.

How passwords are stored

  • Only the hash is stored (UNIX: /etc/shadow, readable only by root).
  • Easy to compute, hard to invert.
  • Salt — added to the password so identical passwords hash differently; defeats rainbow tables.

One-time passwords (OTP)

Both user and server compute the password from a function over a seed; hardware token / calculator / key fob. Changes frequently → no replay possible.

Biometrics

A physical attribute: fingerprint, hand scan, etc.

Multi-Factor Authentication (MFA)

Two or more factors, e.g.: USB dongle + biometric + password.

14. Security defenses

Defense in depth is the key principle: multiple layers of security.

  • Security policy — what exactly is being secured.
  • Vulnerability assessment — comparing reality to the policy.
  • Intrusion Detection (IDS):
    • Signature-based — looks for known bad patterns.
    • Anomaly detection — flags deviations from normal; can catch zero-days.
    • Issues: false positives and false negatives.
  • Virus protection — signature search or sandbox execution.
  • Auditing, accounting, logging of system/network activities.
  • Safe computing — avoid sketchy sources.

Best-practice checklist

  • Educate users: don't attach unknown devices, don't share passwords, beware social engineering, remember that email isn't private.
  • Phishing protection: don't click unknown links/attachments; verify requests by phone.
  • Use secure communication; physically protect hardware.
  • Minimize attack surface — disable unused services.
  • Use modern HW/SW; patch regularly; install only trusted (code-signed) apps.
  • Enable logs + automated alerts.
  • Antivirus, IDS, firewall.
  • Strong passwords; don't write them in obvious places.
  • Encrypt disks and important files.
  • Maintain an up-to-date security policy.

15. Firewalls and security classifications

Firewall

  • Sits between trusted and untrusted hosts.
  • Limits network access between the two security domains.
  • Can be circumvented: tunneling (e.g., telnet inside HTTP), spoofing (rules based on IP/host names that can be forged).
  • Personal firewall — software layer on a host.
  • Application proxy firewall — understands the protocol (e.g., SMTP).
  • System-call firewall — monitors syscalls and applies rules to each program.

Computer Security Classifications (DoD)

The U.S. Department of Defense defines four divisions: A, B, C, D.

ClassWhat it provides
DMinimal security.
CDiscretionary protection via auditing.
C1 — cooperating users at one protection level.
C2 — user-level access control.
BAll of C plus unique sensitivity labels per object. Subdivided into B1, B2, B3.
AFormal design and verification techniques.

16. Example: Windows 10

  • Security is built around user accounts: each user has a unique security ID.
  • Login creates a security access token: user SID, group SIDs, special privileges.
  • Every process receives a copy of the token. The system checks the token on access.
  • A subject model is used: a subject tracks permissions for each program a user runs.
  • Every Windows object has a security descriptor: owner SID, group SID, discretionary ACL, system ACL.
  • Windows added mandatory integrity controls — integrity labels on every securable object/subject.
  • Objects are container objects (e.g., a directory) or noncontainer; by default, objects in a container inherit permissions from the parent.
  • Windows 10 challenges: weak defaults, many bundled services, many installed applications.
Take-away: absolute security is impossible. The aim is to raise the attacker's cost above the expected gain — which requires defense in depth: physical security, safe applications, a well-configured OS, a protected network; cryptography for confidentiality and authentication; thoughtful user authentication (salted passwords, OTP, biometrics, MFA); IDS, antivirus, sandboxing, firewalls; and a constant reminder that the weakest link is usually the human.
